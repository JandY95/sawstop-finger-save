# AGENTS.md

이 문서는 저장소 전체에 적용되는 공통 작업 규칙만 다룬다.
계획 수립 절차, 현재 진행 상태, TODO 목록, 임시 메모는 이 파일에 넣지 않는다.

## 1. 프로젝트 목적

- 이 프로젝트의 1차 목표는 SawStop 사고 접수를 안정적으로 받는 것이다.
- 기본 흐름은 `웹폼 -> Workers -> Notion 사고 DB 저장 -> 첨부는 R2 + 첨부 DB 비동기 처리 -> 같은 사고 페이지 본문에서 영문 리포트 작성/수정 -> 출력 확인 -> 본사 전달`이다.
- 고객은 한국어 웹폼과 접수번호만 본다.
- 1차 완료선은 `본사 전달 가능 상태`까지다.
- 카트리지 분석/결과 회신 단계는 후속 범위다.

- 작업 전 아래 문서를 먼저 확인한다.
- source of truth 우선순위는 아래 순서를 따른다.

- `DECISIONS_LOCK.md`에 잠긴 항목은 추측으로 뒤집지 않는다.
- 하위 문서는 상위 문서를 재정의할 수 없다.
- UI 문서는 UI 범위만 정의한다. 저장/스키마/쓰기 규칙은 상위 문서를 따른다.

## 3. 절대 바꾸면 안 되는 구현 기준

- 고객 입력 채널은 Notion 폼이 아니라 별도 웹폼이다.
- 첨부 원본 저장소는 Notion 파일이 아니라 Cloudflare R2다.
- 운영 기준 레코드는 Notion 사고 페이지다.
- 영문 리포트는 별도 DB가 아니라 같은 사고 페이지 본문에서 관리한다.
- `본문 저장 성공 = 접수 성공`으로 본다.
- 첨부 실패는 접수 실패와 분리한다.
- 손가락 사진은 접수 시 선택이지만 최종 발송 전에는 필수다.
- 1차 발송은 수동 발송이 기본이다.
- 1차 구현 단위는 단일 Workers 프로젝트다.
- 1차 범위에서 첨부 자동 분류를 넣지 않는다.
- 고객 웹폼은 첨부 유형을 묻지 않는다.
- 고객 업로드 첨부는 `첨부 유형 = null`로 생성하고 관리자 뷰에서 후분류한다.
- 관리자 보강 업로드는 업로드 시 첨부 유형을 지정한다.
- 사고 DB의 `첨부(선택)` 파일 속성은 MVP 기준 자동 원본 저장소로 사용하지 않는다.
- `Date of Occurence`에 시간이 없으면 선택 날짜의 `12:00 Asia/Seoul`로 저장한다.
- 첨부 ID 형식은 `ATT-{receiptNumber}-{seq4}`를 유지한다.
- `표시 순서`는 고객 업로드 1,2,3... 순서로 시작하고 관리자 업로드는 현재 최대값 다음부터 이어간다.
- 삭제/휴지통 이동 후에도 표시 순서를 재사용하지 않는다.
- `미리보기 링크`는 MVP에서 R2 접근 URL을 사용한다.
- `썸네일`은 MVP에서 동일 URL 재사용을 허용한다.

## 4. 파일/모듈 수정 원칙

- 한 번에 하나의 흐름만 바꾼다.
- 문서상 책임 경계를 넘는 수정은 묶어서 처리하지 않는다.
- 웹폼, 사고 저장, 첨부 처리, 관리자 보강 업로드, write-back 로직은 책임을 분리한다.
- rollup/formula를 직접 쓰는 코드 경로를 만들지 않는다.
- relation 갱신과 write-back 갱신을 같은 의미로 취급하지 않는다.
- 문서에 없는 새 상태값, 새 자동화, 새 백그라운드 작업, 새 관리자 액션을 임의로 추가하지 않는다.
- 현재 문서에 없는 필드를 편의상 만들지 않는다.
- 루트 `AGENTS.md`에는 repo-wide 규칙만 넣고 구현 계획이나 진행 현황을 넣지 않는다.

## 5. 테스트/검증 우선순위

- 1순위는 스키마 드리프트와 허용값 일치 여부다.
- 2순위는 사고 접수 성공 경로다.
- 3순위는 첨부 0건일 때도 접수가 정상 완료되는지다.
- 4순위는 첨부 일부 실패가 접수 성공을 깨지 않고 `일부 실패`로 분리되는지다.
- 5순위는 관리자 손가락 사진 업로드 후 `손가락 사진 있음`이 갱신되는지다.
- 6순위는 첨부 변경 시 `첨부 최종 확인 완료`가 강제로 false 재평가되는지다.
- 7순위는 `영문 검수 완료`, `첨부 최종 확인 완료`, `출력 확인 완료`가 모두 true일 때 `발송 준비 완료(자동)`이 true가 되는지다.
- UI 수정은 시나리오 검증보다 앞서 문구/필드/순서가 `WEBFORM_UI_SPEC.md`와 일치하는지 확인한다.

## 6. 문서와 코드가 충돌할 때 처리 원칙

- 코드보다 문서를 우선한다.
- 문서끼리 충돌하면 문서 우선순위에 따라 판단한다.
- 잠긴 결정이 있으면 `DECISIONS_LOCK.md`를 우선 적용한다.
- 문서에 없는 동작을 코드에서 발견해도 기존 동작을 source of truth로 승격하지 않는다.
- 충돌을 해소할 근거가 없으면 추측으로 고치지 말고 이슈로 남긴다.

## 7. 임의 기능 추가 금지

- 사용자가 명시하지 않았고 문서에도 없는 기능은 넣지 않는다.
- 특히 아래 항목은 임의 추가 금지다.
- 자동 메일 발송
- AI 기반 첨부 분류
- 별도 영문 리포트 DB
- 고객용 추가 상태 조회 기능
- 추가 관리자 대시보드
- 스키마에 없는 계산 필드/편의 필드
- 휴지통/FIFO/정리 정책의 임의 변경

## 8. 첨부 / R2 / Notion / relation / write-back 핵심 방어선

- 구조화 데이터의 source of truth는 `SAWSTOP 사고 보고`다.
- 첨부 메타데이터와 상태의 source of truth는 `SAWSTOP 첨부 관리`다.
- 첨부 원본 파일의 source of truth는 R2다.
- 운영 on/off의 source of truth는 `SAWSTOP 운영 설정`이다.
- 첨부와 사고의 연결 write 경로는 첨부 DB의 `사고건` relation 하나로 유지한다.
- 사고 DB의 `첨부 목록`은 읽기/집계용으로만 취급한다.
- `R2 Key`에는 최종 `attachments/...` 키만 저장한다.
- `tmp/...` 키를 최종 보관 키처럼 쓰지 않는다.
- formula/rollup 속성에 직접 쓰지 않는다.
- `손가락 사진 있음`은 write-back 대상 체크박스다. formula로 대체하지 않는다.
- `첨부 최종 확인 완료`는 write-back 대상 체크박스다. formula로 대체하지 않는다.
- 첨부 변경 이벤트가 발생하면 두 체크박스를 반드시 재평가한다.
- 재평가 대상 이벤트는 최소 아래를 포함한다.
- 신규 첨부 추가
- 첨부 유형 변경
- 삭제
- 휴지통 이동
- 복원
- FIFO 영구 삭제
- 마지막 손가락 사진 상실
- `손가락 사진 있음 = true` 조건은 `첨부 유형 = 손가락 사진` 이고 `상태 = 현재` 인 첨부가 1개 이상 존재할 때뿐이다.
- 휴지통/영구삭제 첨부는 `손가락 사진 있음` 계산에서 제외한다.
- `첨부 최종 확인 완료 = true` 는 `손가락 사진 있음 = true` 인 경우에만 가능하다.
- 첨부 변화가 있으면 `첨부 최종 확인 완료`는 자동으로 false 재설정 가능해야 한다.
- `발송 준비 완료(자동)`은 아래 3개 완료 상태의 파생 결과로만 다룬다.
- `영문 검수 완료`
- `첨부 최종 확인 완료`
- `출력 확인 완료`

## 9. 웹폼 / UI 작업 시 규칙

- 고객 웹폼이나 관리자 업로드 UI를 수정할 때는 반드시 `docs/source/WEBFORM_UI_SPEC.md`를 먼저 읽는다.
- 화면 순서, 문구, 필수/선택, 에러 문구, 첨부 UX, 성공/실패 화면, 반응형/접근성은 `WEBFORM_UI_SPEC.md`를 따른다.
- UI 편의상 내부 필드를 노출하지 않는다.
- 특히 아래 값은 고객 UI에 노출하지 않는다.
- `page_id`
- `첨부 업로드 상태`
- `손가락 사진 있음`
- `첨부 최종 확인 완료`
- `발송 준비 완료(자동)`
- Queue 상태
- 내부 오류 코드
- 관리자 링크
- UI 문서가 저장 방식까지 바꾸는 근거가 되지 않는다.
- UI 변경이 DB 매핑이나 write-back 규칙을 건드리면 상위 문서를 다시 확인한다.

## 10. 남은 open issue 처리 원칙

- 남아 있는 open issue는 추측으로 닫지 않는다.
- 문서에 `open issue`, `TBD`, `잠금 필요`, `미정`으로 남은 항목은 임의 확정하지 않는다.
- 구현 중 막히면 아래 순서로 처리한다.

1. 상위 문서 재확인
2. `DECISIONS_LOCK.md` 확인
3. 그래도 불명확하면 코드에 임의 규칙을 넣지 말고 issue로 남김

- 특히 아래 유형은 추측 금지다.
- 상태값 전체 목록
- 삭제 사유 옵션
- 본문 블록 구조 상세
- 버튼 부작용
- 자동 날짜 기록 책임 주체
- 오류 플래그 자동 규칙
- 세션 쿠키 정책
- 8시 정리 작업 실제 실행 주체
- FIFO 5GB 측정 기준
- 공식 영문명 사전 저장 위치
- write-back 책임 주체 세부 분담
