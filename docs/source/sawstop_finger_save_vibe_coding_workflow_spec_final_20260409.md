# SAWSTOP Finger Save 바이브코딩용 상세 워크플로우 명세서

## 0. 문서 목적
이 문서는 `SAWSTOP Finger Save 만들기` 프로젝트를 **바이브코딩으로 바로 구현 가능한 수준**까지 풀어쓴 운영/구현 명세서다.

이 문서의 역할은 아래 4가지다.

1. **운영 워크플로우**를 한 번에 복원한다.
2. **웹폼 → Workers → Notion → R2 → 영문 리포트 → 본사 전달** 흐름을 구현 블록 단위로 정리한다.
3. **Notion DB / 속성 / 값 / 매칭 / 검증 / 예외 처리**를 개발자가 바로 따라 만들 수 있게 정리한다.
4. 병준이 원하는 방향, 즉 **고객 편의 우선 / 운영 단순성 / 실무 복원성 / AI와 함께 고치기 쉬운 구조**를 기준으로 설계 원칙을 잠근다.


## 0-1. 2026-04-09 라이브 Notion DB 대조 결과 반영
이 문서는 2026-04-09 기준으로 **라이브 Notion DB를 직접 확인한 뒤** 정정한 버전이다.

이번 대조에서 중요한 점은 아래 3가지였다.

1. `SAWSTOP 사고 보고` DB에는 현재 live schema 기준으로 `첨부 업로드 상태`, `손가락 사진 있음` 속성이 **실제로 반영되어 있다**.
2. `SAWSTOP 첨부 관리` DB는 문서 초안보다 실제 속성이 더 많이 만들어져 있으며, `상태 / 삭제 사유 / 영구삭제 예정 시각 / 휴지통 이동 시각 / 표시 순서 / R2 Key / 미리보기 링크 / 썸네일 / 검수 메모`까지 이미 존재한다.
3. `SAWSTOP 운영 설정` DB에는 `설정명` title과 `고객 접수증 운영 메모`가 실제 속성으로 존재한다.

따라서 이 문서에서는
- **현재 live DB에 실제로 있는 속성**
- **현재 live DB에는 아직 없고, 확장 후보로만 보는 속성**
을 분리해서 적는다.

## 1. 이 프로젝트가 지향하는 점

### 1-1. 가장 중요한 방향
- 고객은 **쉬운 한국어 웹폼**만 보면 된다.
- 병준은 **Notion 중심**으로 사고건을 관리한다.
- 영문 리포트는 **같은 사고 페이지 본문**에서 수정한다.
- 첨부 원본은 Notion file이 아니라 **Cloudflare R2**를 기준 저장소로 쓴다.
- 구조는 처음부터 거창하게 만들지 않고, **수동 기본 + 필요한 블록만 점진 확장**한다.
- 코드 구조는 한 번에 완벽한 정답보다, **블록별로 고치기 쉬운 구조**를 우선한다.

### 1-2. 고객 경험 관점
- 20대부터 60대까지 **PC/모바일에서 큰 설명 없이 입력 가능**해야 한다.
- 사진 첨부는 중요하지만, 접수 단계에서 강제하면 접수 장벽이 올라가므로 **선택 입력**으로 둔다.
- 손가락 사진은 접수 시점이 아니라 **최종 리포트 전까지 반드시 확보**하는 운영 원칙으로 관리한다.
- 고객은 접수 후 복잡한 결과 조회가 아니라 **접수번호만 바로 확인**하면 된다.

### 1-3. 병준 운영 관점
- 같은 사고건 안에서 원문, 첨부, 영문 리포트, 발송 준비 상태를 **한 화면 흐름으로 복원 가능**해야 한다.
- 문제가 생겼을 때 AI에게 “어디가 이상한지”를 바로 넘길 수 있게, **오류 추적 구조가 명확**해야 한다.
- Notion DB 속성은 무작정 text로 뭉개지 않고, **기존 유형 유지 + 저장 직전 변환** 방식을 쓴다.
- 자동화가 실패해도 사고건 자체가 죽지 않게, **본문 저장 성공과 첨부 처리 성공을 분리**한다.

### 1-4. 기술 방향
- 런타임 기준: **Cloudflare Workers + Cloudflare R2 + Notion**
- 입력 채널: **별도 웹폼**
- 저장 작업판: **현재 Notion DB 그대로 사용**
- 배포 방식: **1차는 단일 Workers 프로젝트 1개**
- 발송 방식: **수동 기본**, 회사 SMTP 세팅 완료 후 **반자동 확장**

## 4. 전체 워크플로우

### 4-1. 전체 흐름 한 줄
고객 한국어 웹폼 입력 → Workers 입력 검증 → Notion 사고 DB 저장 + 접수번호 자동 생성 → 첨부 원본 R2 저장 + 첨부 DB 기록 → 병준 1차 검토 → 필요 시 관리자 수동 보완 업로드 → 같은 사고 페이지 본문에서 영문 리포트 초안 작성 → 필요 시 보완 업로드 재진입 → 영문 리포트 최종 수정 → 발송 전 최종 준비(영문 리포트 확인 + 첨부 확인 + 출력본 확인) → 기본은 수동으로 SawStop 본사 메일 발송 → 고객이 활성화된 브레이크 카트리지 발송 → 카트리지 데이터 수집 → 본사 검토 → 최종 결과 회신

### 4-2. 완료선
이 프로젝트의 핵심 워크플로우 완료선은 **본사 전달까지**다.
그 이후의 카트리지 분석과 최종 결과 회신은 **후속 운영 단계**다.

## 5. 핵심 운영 워크플로우 상세

### 5-1. 1단계: 고객 접수
고객은 공개 웹폼에서 사고 내용을 입력한다.

핵심 원칙:
- 웹폼은 한국어 한 페이지 구역형이다.
- 필수 항목만 최소화한다.
- 사진 첨부는 없어도 접수 가능하다.
- Workers가 검증 후 `접수번호`를 먼저 만들고, Notion 사고 DB에 본문을 저장한다.
- 첨부가 있으면 R2와 첨부 DB 기록이 이어진다.

성공 판정 기준:
- **본문 저장 성공 = 접수 성공**
- 첨부 일부 실패, 관리자 알림 실패, 고객 접수증 메일 실패는 접수 실패로 보지 않는다.

### 5-2. 2단계: 병준 1차 검토
병준은 Notion 사고건을 열어 아래를 확인한다.
- 입력값 누락 여부
- 첨부 존재 여부
- 손가락 사진 확보 여부
- 카트리지 시리얼 확인 가능 여부
- 영문화에 필요한 기본 원천 데이터가 들어왔는지

핵심 원칙:
- 손가락 사진은 접수 시점 필수 아님
- 하지만 **최종 리포트 전에 반드시 확보**

### 5-3. 3단계: 필요 시 관리자 수동 보완 업로드
후속으로 받은 손가락 사진, 카트리지 사진, 잘못 올린 사진 교체를 처리한다.

핵심 원칙:
- 공개 고객 웹폼과 분리된 **별도 관리자 업로드 페이지** 사용
- 같은 도메인 숨은 경로 1개
- 비밀번호 1개 + Turnstile + session cookie 기반 인증 유지
- 5회 실패 시 10분 잠금
- 완료건 검색 제외
- 접수번호 자동 포함 링크 지원

### 5-4. 4단계: 영문 리포트 초안 작성
영문 리포트는 별도 DB가 아니라 **같은 사고 페이지 본문**에서 직접 작성한다.

핵심 원칙:
- 템플릿 항목명은 Notion DB의 영문 속성명을 그대로 쓴다.
- 본문 안에 영문 리포트 섹션이 존재한다.
- 병준이 같은 페이지에서 원문과 영문 리포트를 함께 보고 수정할 수 있어야 한다.

### 5-5. 5단계: 필요 시 관리자 보완 업로드 재진입
영문 초안 작성 후에도 첨부가 부족하면 다시 관리자 업로드로 들어간다.

핵심 원칙:
- 리포트 초안 작성 후에도 첨부 보완이 가능해야 한다.
- 첨부 보완 후 다시 같은 페이지 본문으로 돌아와 수정한다.

### 5-6. 6단계: 영문 리포트 최종 수정
보완 첨부를 반영해 발송 직전 본문을 확정한다.

### 5-7. 7단계: 발송 전 최종 준비
아래 3개가 모두 true여야 한다.
- 영문 리포트 확인
- 첨부 확인
- 출력본 확인

이 3개가 모두 끝나야 `발송 준비 완료(자동)`가 true가 된다.

### 5-8. 8단계: 본사 전달
기본 운영은 **수동 발송**이다.
회사 SMTP 세팅 완료 후에만 반자동 확장을 연다.

### 5-9. 9단계: 후속 검토 및 결과 회신
- 고객이 활성화된 브레이크 카트리지를 발송
- 카트리지 데이터 수집
- 본사 검토
- 최종 결과 회신

이 단계는 앱의 1차 완료 판정 바깥이다.

- 영문 리포트 본문 수정
- 발송 준비 상태 확인

#### R2
- 원본 첨부 저장소
- tmp → attachments 이동 구조
- FIFO / 휴지통 운영의 실제 객체 저장소

## 8. DB 1: SAWSTOP 사고 보고

### 8-2. 2026-04-09 기준 live DB에서 직접 확인된 실제 속성

### 8-3. 현재 live DB 기준 추가 반영이 끝난 핵심 속성
아래 2개는 현재 문서/소스/구현 계획에서 중요하게 쓰는 속성이며, **2026-04-09 기준 live Notion DB에도 실제 반영 완료** 상태다.

- `첨부 업로드 상태` (select)

### 9-1. 역할

### 9-4. 현재 문서 초안에서 권장이라고 적었지만 live DB에는 아직 없는 후보 속성
아래는 **현재 live DB에는 없고**, 추가 여부를 따로 판단해야 하는 후보들이다.

- `출처` (`고객 웹폼` / `관리자 수동 보완`)
- `최근 변경 이력`
- `mime type`
- `file size`
- `hash`
- `접수번호` 복제 속성

주의:
- `원본 파일명`이라는 이름은 현재 DB에는 없고, 실제 속성명은 `파일명`이다.
- `R2 key`라는 표기보다 실제 속성명은 `R2 Key`다.
- `원본 상태`라는 별도 text/select 속성은 현재 없고, 실제 상태 관리는 `상태` status + 날짜/삭제 사유 조합으로 이미 운영되고 있다.

### 9-5. 이 DB의 핵심 운영 원칙
- 첨부 분류는 1차 범위에서 AI 자동분류를 쓰지 않는다.
- 병준이 직접 `첨부 유형`을 지정한다.
- 손가락 사진은 최종 리포트 전 반드시 확보한다.
- 후속으로 받은 사진도 똑같이 **R2 + 첨부 DB + 사고건 relation** 구조로 넣는다.
- 휴지통과 FIFO는 이 DB 상태와 R2 원본 상태를 함께 기준으로 처리한다.

## 14. 웹폼 검수/검증 계층

### 14-1. 검증은 3단계로 본다
#### A. 프론트 즉시 검증
- 필수값 누락
- 이메일 형식
- 전화번호 1차 형식
- 기계 시리얼 형식 1차 검사
- 첨부 형식/용량/개수 검사

#### B. 서버 최종 검증
- Turnstile 검증
- 필수값 다시 확인
- 정규화 후 형식 재검사
- MIME type / 파일 크기 재검사
- select/multi_select 매핑값 재검사

#### C. 저장 전 Notion 유형 변환 검증
- text로 넣을 값인지
- select 옵션명과 정확히 일치하는지
- multi_select 배열이 허용값과 일치하는지
- date로 합쳐질 값이 유효한지

### 14-2. 중요한 원칙
- 웹폼 입력 유형과 Notion 저장 유형은 꼭 같을 필요 없다.
- 웹폼은 UX를 담당하고, 저장 직전에 Notion 유형에 맞게 변환한다.
- 자유도가 큰 값은 text 저장이 유리하다.
- 고정 목록은 select/multi_select를 유지한다.

예:
- `Type of Material Being Cut?` → text 유지
- `Consent for Promotional Use` → select 유지
- `Were There Other Devices Being Used When the Cut was Made?` → multi_select 유지

## 17. Workers 제출 처리 흐름

### 17-1. 고정 흐름
`GET /` → 고객 웹폼 HTML 서빙

`POST /submit` 처리 순서:
1. content-type 확인
2. formData 파싱
3. Turnstile 검증
4. normalize
5. validate
6. 접수번호 생성
7. Notion 본문 저장
8. 성공 응답 반환
9. waitUntil로 메일/Queue 처리

### 17-2. 공식 제출 흐름 문장
`Turnstile 확인 → 입력값 검증 → Workers에서 접수번호 생성 → Notion 본문 저장 → page_id 확보 → 고객 성공 응답 반환 → 관리자 알림/고객 접수증 메일 비동기 시도 → Queue 적재 → 큐 소비자 첨부 순차 처리`

### 17-3. 성공 조건
- Notion 본문 저장 성공
- 상태는 `접수`
- 고객에게는 `접수 성공 여부 + 접수번호`만 전달
- `page_id` 같은 내부값은 노출하지 않음

## 18. 비동기 첨부 처리 구조

### 18-1. 업로드 원칙
- 최대 4장
- 형식: `jpg / jpeg / png / webp / heic / heif`
- 파일당 10MB 이하
- 원본 파일명은 그대로 허용
- 내부 저장 키만 별도로 생성

### 18-2. 저장 키 규칙
- 임시: `tmp/{접수번호}/{순번}_{업로드시각}.{확장자}`
- 본경로: `attachments/{접수번호}/{순번}_{업로드시각}.{확장자}`
- 순번은 재사용하지 않음

### 18-3. Queue 메시지
- 접수건 1건당 메시지 1개
- 경량 payload만 담음
- 포함값:
  - `접수번호`
  - `page_id`
  - `첨부 개수`
  - 첨부별 참조 정보
  - `재시도 횟수`

### 18-4. Consumer 처리 순서
첨부 1장씩 아래 순서:
1. 검증
2. 임시 저장 확인
3. 본경로 업로드
4. 첨부 DB 기록
5. relation 연결
6. 사고 DB `첨부 업로드 상태` 갱신

### 18-5. 재시도 규칙
- 파일별 최대 2회 자동 재시도
- 그래도 실패하면 관리자 수동 보완 업로드 대상으로 넘김
- 고객이 다시 올리지 않아도 접수는 유지

### 18-6. `첨부 업로드 상태` 갱신 규칙
- 첨부 0개: `완료`
- 첨부 1개 이상: 본문 저장 직후 `처리중`
- 모두 성공: `완료`
- 일부 실패: `일부 실패`
- 전부 실패: `실패`

## 20. 휴지통 / FIFO / 상태 동기화

### 20-1. `손가락 사진 있음`
현재 첨부 목록 안에서
- `첨부 유형 = 손가락 사진`
- `상태 = 현재`
인 파일이 1개 이상이면 true.

아래 상태는 계산에서 제외:
- `휴지통`
- `영구삭제` (FIFO 포함)
- 그 외 삭제 처리된 항목

### 20-2. `첨부 최종 확인 완료`
- 사람의 최종 검토 체크
- 손가락 사진이 있어야 true 가능
- 아래 상황이면 자동 false 해제
  - 새 첨부 추가
  - 유형 변경
  - 삭제
  - 휴지통 이동/복구
  - 영구삭제 처리(FIFO 포함)
  - 마지막 손가락 사진 상실

### 20-3. 휴지통
- 7일 복구 가능
- 개별 첨부 단위
- 원래 사고건으로만 복구 가능
- 다음 오전 8시 첨부 정리 스케줄에서 만료 삭제

### 20-4. FIFO
- 휴지통 만료 삭제 먼저
- 그래도 5GB 초과면 FIFO 적용
- FIFO 삭제는 휴지통으로 보내지 않음
- live DB 기준 첨부 상태는 `영구삭제`로 정리하고, 필요하면 `삭제 사유` 또는 `검수 메모`에 FIFO 삭제 이력을 남긴다.
- 마지막 손가락 사진이 사라져도 별도 팝업 없이 상태값만 다시 계산

### 24-1. 출력 원본
- 웹뷰와 PDF는 **같은 영문 리포트 원본 본문**을 기준으로 한다.

### 24-2. 디자인 기준
- GitHub Docs 계열 기술 문서형
- 흰 배경
- 아주 옅은 청회색 섹션 배경
- 짙은 네이비 제목
- 짙은 회색 본문
- 산세리프 계열
- 자동 줄바꿈
- 값은 항목명 아래 줄
- 여백 중심 레이아웃
- 과도한 장식 금지
- 모바일 반응형 유지

### 24-3. 첨부 이미지 출력
- 사진 최대 4장
- 출력에서는 가독성 우선
- 카트리지 시리얼 사진은 방향 보정 허용
- 자동 판독은 신뢰도 충분할 때만 문서 값 반영

## 25. 발송 운영 기준

### 25-1. 기본값
- 본사 발송 기본값: `수동`
- 반자동은 SMTP 세팅 완료 후 확장

### 25-2. 사고 DB 발송 관련 속성 의미
- `영문화 모드`: 영문 출력 방식 선택
- `발송 방식`: `수동 / 반자동`
- `영문 검수 완료`: 영문 본문 수정 완료
- `출력 확인 완료`: PDF/웹뷰 최종 확인 완료
- `첨부 최종 확인 완료`: 손가락 사진 포함 첨부 최종 검수 완료
- `발송 준비 완료(자동)`: 위 3개 true면 자동 true
- `발송 완료 시각`: 실제 본사 발송 시점
- `발송 실패 메모`: 실패 사유 기록

### 25-3. 발송 상태 판단
별도 `발송 상태` 속성은 1차에 만들지 않는다.
아래 조합으로 본다.
- `발송 준비 완료(자동)`
- `발송 완료 시각`
- `발송 실패 메모`

## 26. 메일 관리 블록 기준

### 26-1. 이 블록의 위치
- 앱 완성 후 확장 블록
- 1차 핵심 접수 성공 조건과는 분리

### 26-2. 서비스 우선순위
- `1차 Daum`
- `2차 Naver`
- `3차 Gmail`
- `Outlook 마지막 보류`

### 26-3. 1차 구현 원칙
- Daum만 실제 활성화
- Naver/Gmail/Outlook은 프리셋만 유지

### 26-4. 저장/테스트 원칙
- `저장`과 `테스트 발송` 분리
- 비밀번호는 저장하지 않음
- 저장은 현재 비밀번호로 1회 인증 성공 시에만 완료
- 테스트 발송은 현재 화면값 기준으로 바로 실행 가능
- 테스트 성공이 저장을 대신하지 않음

### 26-5. SMTP 완료 판정
아래 3개를 모두 통과해야 함.
- 서비스 보안 전제 충족
- 테스트 송수신 1회 성공
- 실제 운영 계정 권한 확인 완료

## 27. 오류 처리 기준

### 27-1. 1차 원칙
- `서버/호스팅 로그 + 관리자 이메일 알림`
- Notion 오류 DB, 문자, 카카오톡은 1차 제외

### 27-2. 고객 화면 실패 문구
`접수가 완료되지 않았습니다. 잠시 후 다시 시도해 주세요.`

### 27-3. 고객 화면 원칙
- 관리자 추적용 오류코드 노출 안 함

### 27-4. 관리자 오류 메일
#### 제목
`SawStop 웹폼 접수 오류 알림`

#### 첫 줄
`SawStop 웹폼 접수 중 오류가 발생했습니다.`

#### 본문 순서
1. 오류코드
2. 발생 시간
3. 문제가 발생한 위치
4. 관련 항목
5. 오류 원인
6. 오류 메시지

#### 마지막 문구
- `오류 원인을 확인하고 조치해 주세요.`
- `필요 시 서버 로그를 함께 확인해 주세요.`

### 27-5. `문제가 발생한 위치` 값
- `입력값 확인`
- `파일 업로드`
- `Notion 저장`
- `이메일 알림`

### 27-6. 오류 원인 예시
- `필수값 누락 (필수 항목이 비어 있음)`
- `이메일 형식 오류 (example@domain.com 형식 아님)`
- `시리얼 번호 형식 오류 (C/P/I + 숫자 9자리 형식 아님)`
- `첨부 파일 용량 초과 (허용 용량 초과)`
- `Notion 옵션값 불일치 (select 또는 multi_select 값 매칭 실패)`
- `Notion API 응답 지연 (저장 요청 시간 초과)`
- `이메일 서버 인증 실패 (SMTP 인증 오류)`

### 27-7. 오류 메시지 원칙
- 가능한 한 시스템 원문 그대로 남김
- 예:
  - `Email format is invalid`
  - `Request timeout after 10000ms`
  - `Value is not a valid select option`
  - `File size exceeds limit`

## 28. 고객 완료 화면 기준

고객에게는 복잡한 내부 상태를 보여주지 않는다.

### 28-1. 성공 시 표시
- `접수가 완료되었습니다.`
- `입력하신 내용은 내부 검토 후 확인됩니다.`
- `접수번호: {접수번호}`

### 28-2. 첨부 상태 노출 원칙
- 접수번호 표시가 핵심
- 고객 완료 화면에는 첨부 상태 요약을 기본 노출하지 않는다.
- 첨부 상태는 내부 노션에서 관리 우선으로 본다.
- 첨부 일부 실패/실패는 내부 검토와 후속 보완 업로드로 처리한다.

## 31. 개발자가 바로 구현할 때 꼭 지켜야 할 것

### 31-1. 절대 흔들면 안 되는 기준
- 고객 입력은 Notion 폼이 아니라 **별도 웹폼**
- 첨부 원본 기준 저장소는 **R2**
- 사고건 기준 작업판은 **Notion 사고 페이지**
- 영문 리포트는 **같은 사고 페이지 본문**
- 접수 성공 기준은 **본문 저장 성공**
- 첨부 실패는 **접수 실패와 분리**
- 손가락 사진은 **접수 단계 선택, 최종 리포트 전 필수 확보**
- 1차 발송은 **수동 기본**
- 1차 구현은 **단일 Workers 프로젝트**

### 31-2. 옵션명이 꼬이면 안 되는 항목
- `Consent for Promotional Use`
- `Was There A Visible Injury Mark?`
- `Type of blade being used`
- `Was the saw operator wearing gloves at the time?`
- `What was the approximate feed rate ...`
- `Were There Other Devices Being Used When the Cut was Made?`
- `첨부 업로드 상태`
- `첨부 유형`
- `영문화 모드`
- `발송 방식`

### 31-3. AI에게 수정 요청할 때 좋은 방식
- 접수번호 예시
- 실패한 단계 (`입력값 확인/파일 업로드/Notion 저장/이메일 알림`)
- 실제 오류 메시지 원문
- 화면에서 입력한 값 예시
- 기대 동작 / 실제 동작

이 5가지를 같이 넘기면 AI가 고치기 쉬워진다.

## 33. 한 줄 결론
이 프로젝트의 핵심은 **고객은 쉬운 한국어 웹폼으로 접수하고, 병준은 Notion 사고 페이지를 중심으로 첨부·영문 리포트·발송 준비를 관리하며, 실제 시스템은 Workers + R2 + Notion이 느슨하게 연결된 복구 쉬운 블록 구조로 운영한다**는 점이다.
