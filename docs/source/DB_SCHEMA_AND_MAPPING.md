## 1. 문서 개요와 사용 원칙

### 1-1. 이 문서가 잠그는 것
이 문서는 아래 항목을 구현 전에 먼저 잠근다.

1. 시스템 간 데이터 이동 경로
2. Notion 속성명 / 타입 / 허용값 / 저장 시점
3. 웹폼 입력값의 정규화 및 서버 검증 기준
4. 접수 성공 기준과 첨부 실패 분리 원칙
5. 첨부 메타데이터 저장 구조와 R2 연결 구조
6. relation / rollup / formula / button 사용 전제와 제약
7. 관리자 수동 보완 업로드의 검색·인증·저장 규칙
8. 구현 전에 반드시 닫아야 할 open issue

### 1-2. 이 문서가 잠그지 않는 것
아래 항목은 이 문서의 직접 범위가 아니다.

- UI 상세 디자인
- 영문 리포트 본문 문장 품질
- PDF 스타일 세부값
- SMTP 실제 계정 설정값
- 후속 카트리지 분석 운영

### 1-3. 금지 사항
- live schema에 없는 속성을 “있다고 가정”해서 구현하지 않는다.
- 후보 속성을 실제 저장 로직의 전제로 쓰지 않는다.
- 사고 DB의 파생값을 사람 손으로 다시 입력해 이중 원천을 만들지 않는다.
- 첨부 원본을 Notion files만 기준 저장소로 삼지 않는다.
- 첨부 실패를 접수 실패로 승격하지 않는다.
- 문서에 없는 상태값, select 옵션, multi_select 옵션을 임의 생성하지 않는다.
- 별도 영문 리포트 DB, 별도 발송 상태 DB, 별도 첨부 원본 DB를 임의로 만들지 않는다.

### 1-4. live schema와 코드가 다를 때 처리 원칙
1. **live DB 실제 속성**이 우선이다.
2. 문서 초안에만 있고 live DB에 없는 항목은 후보로만 유지한다.
3. 코드가 live schema와 다르면 코드가 아니라 schema/문서/매핑을 먼저 재대조한다.
4. drift가 확인되면 추측으로 보정하지 말고 이 문서의 `open issue`에 올린 뒤 확정 후 반영한다.

### 1-5. 이 문서에서 사용하는 용어
- **사고 DB**: `SAWSTOP 사고 보고`
- **첨부 DB**: `SAWSTOP 첨부 관리`
- **운영 설정 DB**: `SAWSTOP 운영 설정`
- **live 속성**: 2026-04-09 기준 원문에서 실제 존재가 확인된 속성
- **후보 속성**: 문서상 언급되었지만 live DB에는 아직 없다고 명시된 속성
- **write-back**: formula가 아닌 속성에 대해 시스템 처리 로직이 값을 다시 써 넣는 것

### 1-6. 2026-04-11 시니어 엔지니어 잠금 결정
아래 항목은 이 개정판에서 더 이상 open issue가 아니라 **구현 잠금 결정**으로 본다.

1. `Date of Occurence` 시간 미상 입력은 **선택한 날짜의 12:00 (Asia/Seoul)** 로 저장한다. 별도 live 속성이 없으므로 date 단일값은 이 규칙으로 고정한다.
2. 사고 DB `첨부(선택)` file 속성은 **코드의 원본 저장소/자동 write 대상이 아니다.** 1차 MVP의 첨부 원천은 `R2 + 첨부 DB + 사고 DB relation` 구조로 고정한다.
3. Queue payload는 **경량 고정 스키마**를 사용한다. 접수건당 메시지 1개, 첨부별 참조 정보는 `seq / tmpKey / originalFileName / contentType / sizeBytes`까지로 제한한다.
4. `첨부 ID`는 **`ATT-{receiptNumber}-{seq4}`** 형식으로 고정한다. 예: `ATT-202604031224-2839-0001`
5. `표시 순서`는 **고객 웹폼 선택 순서 그대로 1,2,3...** 를 사용하고, 관리자 보완 업로드는 **현재 최대값 + 1** 부터 이어 붙인다. 삭제/휴지통/복구/FIFO 후에도 순번은 재사용하지 않는다.
6. `미리보기 링크`는 **R2 접근 URL(서명 URL 또는 Worker 프록시 URL)** 로 저장하고, 1차 MVP에서 `썸네일`은 **같은 URL 재사용을 허용**한다. 별도 썸네일 파이프라인은 1차 범위에 넣지 않는다.
7. 고객 웹폼 첨부는 업로드 시점에 `첨부 유형`을 묻지 않는다. **첨부 DB 생성 시 `첨부 유형`은 빈값(null)으로 저장**하고, `첨부 분류 대기` 뷰에서 병준이 수동 분류한다. 관리자 보완 업로드는 업로드 시점에 유형을 즉시 선택한다.
8. 사고 DB `상태` 옵션은 **`접수 / 진행중 / 완료 / 반려`** 로 잠근다.
9. 첨부 DB `상태` 옵션은 **`현재 / 휴지통 / 영구삭제`** 로 잠근다.
10. 첨부 DB `삭제 사유`는 property 이름 **`삭제 사유`**, type **`select`**, 옵션 **`화질 불량 / 기타 / 불필요 / 오업로드 / 중복`** 으로 잠근다.
11. 실제 사고 페이지 본문 block 구조는 `Report a Save (Known or Suspected Finger Contact)`부터 `Attachments`까지의 확인된 heading 순서와 마지막 `첨부(선택):` 후 빈 블록 1개로 잠근다.
12. 관리자 인증은 Cloudflare Workers Secrets `ADMIN_PASSWORD`, `ADMIN_SESSION_SECRET` 기반의 비밀번호 로그인 + 서명된 세션 쿠키 + 로그인 실패 잠금으로 잠그며, 현재 관리자 로그인에는 Turnstile을 적용하지 않는다.
13. 첨부 원본 저장소 기준은 **R2**로 유지한다.
14. 휴지통 첨부는 **7일 복구 가능**하며, 만료 휴지통 정리는 FIFO보다 먼저 적용된다.
15. FIFO는 만료 휴지통 정리 후에도 저장소가 **5GB 초과**인 경우에만 적용될 수 있다.
16. FIFO 삭제는 휴지통을 거치지 않고 첨부 DB row를 **`영구삭제`** 상태로 정리한다.
17. `check:fifo-trash-candidates` 같은 live-read 검증은 deterministic parity, scenario execution, baseline, CI, product wiring과 분리된 수동 확인 경계로 유지한다.

## 2. 시스템 데이터 모델 개요

### 2-1. 핵심 객체

| 객체 | 역할 | 단일 진실 원천(source of truth) | 비고 |
|---|---|---|---|
| 웹폼 제출 데이터 | 고객 입력 원문 수집 | Workers 정규화 직전 form-data | 저장 전 단계 |
| 사고건 | 사고 1건의 중심 레코드 | `SAWSTOP 사고 보고` DB 페이지 1개 | 영문 리포트 본문도 같은 페이지 |
| 첨부 메타데이터 | 첨부 1개 단위의 상태/분류/삭제/표시 추적 | `SAWSTOP 첨부 관리` DB 1행 | 첨부 1개 = 1행 |
| 첨부 원본 파일 | 실제 원본 이미지 객체 | Cloudflare R2 | tmp → attachments 이동 구조 |
| 운영 설정 | 관리자 알림/고객 접수증 on/off 및 운영 메모 | `SAWSTOP 운영 설정` DB 1행 | 비밀값 제외 |

### 2-2. 시스템 간 데이터 흐름
1. 고객이 웹폼에 입력한다.
2. Workers가 form-data를 파싱하고 정규화/검증한다.
3. Workers가 `접수번호`를 생성한다.
4. Workers가 사고 DB 페이지를 먼저 저장한다.
5. 고객에게 성공 응답과 접수번호를 반환한다.
6. 이후 비동기에서 Queue/Consumer가 첨부를 처리한다.
7. 첨부 원본은 R2에 저장한다.
8. 첨부 메타데이터는 첨부 DB에 기록한다.
9. 사고 DB는 relation / rollup / formula / checkbox write-back을 통해 후행 상태를 반영한다.

### 2-3. 객체 간 연결 구조
- 사고건 1건 ↔ 첨부 N개 = `사고 DB 1페이지` ↔ `첨부 DB N행`
- 첨부 DB 각 행은 `사고건` relation으로 원래 사고건에 연결된다.
- 사고 DB의 첨부 개수 관련 값은 직접 숫자 입력이 아니라 rollup 기반이다.
- 발송 준비 여부는 사람 검토용 checkbox 3개와 formula를 조합해 판단한다.
- 운영 설정은 별도 relation 없이 Workers가 단일 설정 1행을 직접 읽는다.

### 2-4. 단일 진실 원천 정리
- 고객 원문 사고 정보: `SAWSTOP 사고 보고`
- 첨부 분류/삭제/휴지통/FIFO 메타데이터: `SAWSTOP 첨부 관리`
- 첨부 원본 존재 및 저장 경로: `R2`
- 무료 알림/접수증 on/off: `SAWSTOP 운영 설정`
- 접수 식별자: 사고 DB의 title인 `접수번호`
- 첨부 관련 파생 숫자: 사고 DB rollup
- 발송 가능 여부의 파생 판단: 사고 DB `발송 준비 완료(자동)` formula

### 2-5. 구현 방어선: 쓰기 경로 고정
원문은 relation이 존재한다고만 적고, 코드가 어느 쪽 relation을 쓰는지까지는 확정하지 않는다. drift 방지를 위해 이 문서는 아래를 구현 방어선으로 고정한다.

- **코드의 relation 쓰기 경로는 첨부 DB `사고건` relation 1곳으로 고정한다.**
- 사고 DB `첨부 목록`은 그 반대 relation을 조회하는 읽기 기준으로 본다.
- rollup / formula / button은 원천 데이터 쓰기 경로가 아니다.

이 규칙은 새 기능 추가가 아니라, 같은 relation을 코드 곳곳에서 양방향으로 쓰면서 생기는 정합성 위험을 줄이기 위한 방어선이다.

- 첨부 relation 집계
- 발송 준비 상태 관리

### 4-3. 사고 보고 DB 후보 속성
원문에는 사고 보고 DB의 추가 후보 속성이 별도로 제시되지 않았다.  
따라서 이 문서에서는 **사고 보고 DB 후보 속성을 임의로 추가하지 않는다.**

### 4-4. 사고 보고 DB 핵심 규칙 요약
- 신규 접수 성공 기준은 **사고 DB 본문 저장 성공**이다.
- `접수번호`는 title이며 시스템 공통 키다.
- `첨부 업로드 상태`는 접수 성공/실패가 아니라 **후행 첨부 처리 결과**를 나타낸다.
- `손가락 사진 있음`, `첨부 최종 확인 완료`, `발송 준비 완료(자동)`은 모두 발송 전 단계 품질 관리용이다.
- 영문 리포트는 별도 DB가 아니라 **같은 사고 페이지 본문**에서 작성한다.

### 4-5. 사고 페이지 본문 block 구조
실제 사고 페이지 본문 block 구조는 아래 순서로 잠근다.

1. `Report a Save (Known or Suspected Finger Contact)`
2. `Incident Information`
3. `People / Contact Information`
4. `Injury Information`
5. `Saw / Cartridge Information`
6. `Material / Setup / Conditions`
7. `Incident Description`
8. `Attachments`
9. 마지막에 `첨부(선택):` 후 빈 블록 1개

이 구조는 본문 저장 성공 판정과 운영 검수 흐름의 기준이다.

- AI 자동분류를 쓰지 않는다.
- 첨부 유형은 병준이 직접 지정한다.
- 후속 보완 첨부도 동일하게 **R2 + 첨부 DB + 사고건 relation** 구조로 넣는다.
- 휴지통과 FIFO는 **첨부 DB 상태 + R2 원본 상태**를 함께 기준으로 본다.
- `원본 파일명`은 현재 속성명이 아니며 실제 속성명은 `파일명`이다.
- `R2 key`가 아니라 실제 속성명은 `R2 Key`다.
- 첨부 DB `R2 Key`에는 **최종 attachments 경로**만 저장한다.

## 8. 사고 DB ↔ 첨부 DB relation / rollup / formula / button 사용 방식과 제약

### 8-1. relation 사용 방식
- 사고 DB `첨부 목록`은 첨부 DB 행들을 가리키는 relation이다.
- 첨부 DB `사고건`은 반드시 원래 사고건 1개에 연결된다.
- 첨부 복구는 **원래 사고건으로만** 가능하다.
- 첨부 개수와 손가락 사진 유무 같은 파생 판단은 relation 기반 첨부 집합에서 계산한다.

#### relation 구현 제약
- relation 연결 실패 시 첨부 원본이 R2에 있어도 사고건 화면에서 누락될 수 있다.
- relation 자체를 수동으로 임의 수정하면 rollup/배지가 왜곡될 수 있다.
- relation은 파생값의 입력 원천이므로, 첨부 수를 다른 숫자 속성으로 직접 관리하면 drift가 생긴다.
- **코드 쓰기 경로는 첨부 DB `사고건` relation 1곳으로 고정한다.**
- 사고 DB `첨부 목록`을 코드가 별도 진실 원천처럼 직접 관리하지 않는다.

### 8-2. rollup 사용 방식
사고 DB의 아래 3개 값은 rollup이다.

- `현재 첨부 개수`
- `휴지통 첨부 개수`
- `전체 첨부 개수`

이 값들은 첨부 DB의 relation + formula/상태 기반 집계를 반영한다.

#### rollup 구현 제약
- 직접 저장/수정 대상이 아니다.
- relation과 첨부 DB 값이 맞아야만 정확하다.
- rollup 숫자를 코드가 “주 원천”으로 삼지 않는다.
- 화면 표시나 빠른 배지 표시에는 사용할 수 있지만, 최종 판단이 필요할 때는 원천 relation/첨부 DB 상태를 다시 확인할 수 있어야 한다.
- API 조회 시 축약이나 pagination 이슈 가능성을 고려해야 한다.

### 8-3. formula 사용 방식
사고 DB의 대표 formula는 다음이다.

- `발송 준비 완료(자동)`

첨부 DB의 대표 formula는 다음이다.

- `복구 가능`
- `현재 카운트용`
- `휴지통 카운트용`
- `전체 카운트용`

#### formula 구현 제약
- formula 값은 직접 업데이트 대상이 아니다.
- formula는 입력값이 아니라 **파생 판단 결과**로만 사용한다.
- formula가 참조하는 원천 속성이 흔들리면 파생값도 동시에 틀어진다.
- `손가락 사진 있음`과 `첨부 최종 확인 완료`는 formula가 아니라 checkbox이므로, formula처럼 자동 계산된다고 가정하면 안 된다.

### 8-4. button 사용 방식
사고 DB의 버튼은 아래 3개다.

- `접수->진행중`
- `접수->반려`
- `진행중->완료`

#### button 구현 제약
- button은 **운영용 UI 전환 도구**로 본다.
- 이 문서 기준으로 button의 내부 액션 정의는 원문에 상세히 적혀 있지 않다.
- 따라서 구현 로직이 button 클릭 자체를 API 전제 조건으로 삼아서는 안 된다.
- 상태 판단의 진실 원천은 button 이름이 아니라 실제 `상태` 값과 관련 날짜/검수 값이다.
- button이 상태 외에 무엇을 함께 수정하는지는 open issue로 남긴다.

### 8-5. source of truth 구분
- 첨부 개수 표시: rollup은 **표시용 파생값**
- 첨부 현재/휴지통/영구삭제 상태: 첨부 DB `상태` + 관련 날짜
- 손가락 사진 있음: 첨부 DB `첨부 유형` + `상태` 집합을 기준으로 시스템이 checkbox write-back
- 발송 준비 완료(자동): 사고 DB checkbox 3개에서 formula로 계산
- 상태 전환 button: 원천 아님, 보조 UI

### 9-1. 형식
`YYYYMMDDHHmm-연락처뒷4자리`

예시: `202604031224-2839`

### 9-2. 의미

- Workers가 생성한다.
- 사고 DB title로 저장한다.
- 첨부 저장 경로, Queue payload, 로그 모두 이 값을 공통 키로 사용한다.
- 전화번호 정규화가 선행되어야 뒷4자리 추출이 안정적이다.

## 10. 접수 성공 기준과 실패 분리 원칙

### 10-1. 접수 성공 기준
아래가 충족되면 접수 성공이다.

- Turnstile 확인
- 입력값 검증 통과
- 접수번호 생성 성공
- Notion 사고 DB 본문 저장 성공
- 상태 `접수`로 시작

즉, **본문 저장 성공 = 접수 성공**이다.

### 10-2. 접수 실패로 보지 않는 항목
아래는 실패해도 접수 실패로 보지 않는다.

- 첨부 일부 실패
- 첨부 전체 실패
- 관리자 알림 실패
- 고객 접수증 메일 실패

### 10-3. 실패 분리 이유
- 사고건 자체가 죽지 않게 하기 위함
- 첨부는 후속 보완 업로드로 복구 가능
- 관리자 알림/고객 접수증은 비동기 부가 기능이기 때문

### 10-4. 고객 노출 원칙
- 고객에게는 `접수 성공 여부 + 접수번호`만 보여준다.
- 내부값인 `page_id`는 노출하지 않는다.
- 첨부 상태는 고객 완료 화면 기본 노출 대상이 아니다.

## 11. 첨부 업로드 상태 갱신 규칙

### 11-1. 사고 DB `첨부 업로드 상태`
허용값은 아래 4개만 사용한다.

- `처리중`
- `완료`
- `일부 실패`
- `실패`

### 11-2. 갱신 규칙

| 조건 | 저장값 |
|---|---|
| 첨부 0개 | 완료 |
| 첨부 1개 이상이며 사고 DB 본문 저장 직후 | 처리중 |
| 첨부 전부 성공 | 완료 |
| 일부 성공 / 일부 실패 | 일부 실패 |
| 전부 실패 | 실패 |

### 11-3. 갱신 주체와 시점
- 신규 접수 직후 초기값 설정: Workers
- 후행 첨부 결과 반영: Queue/Consumer
- 관리자 수동 보완 업로드 후 재판단: 관리자 업로드 처리 로직

### 11-4. 실패 영향
- 첨부 상태 실패는 접수 실패가 아니다.
- 내부 노션에서 확인 후 보완 업로드로 복구한다.

## 12. 손가락 사진 있음 / 첨부 최종 확인 완료 / 발송 준비 완료(자동) 규칙

### 12-1. 손가락 사진 있음
true 조건:
- 현재 첨부 목록 안에서
- `첨부 유형 = 손가락 사진`
- `상태 = 현재`
인 파일이 1개 이상

계산 제외:
- `휴지통`
- `영구삭제`
- 그 외 삭제 처리 항목

#### 구현 제약
- 이 속성은 formula가 아니라 checkbox이므로 시스템 write-back이 필요하다.
- 아래 이벤트 후 반드시 재판단해야 한다.
  - 새 첨부 추가
  - 첨부 유형 변경
  - 삭제
  - 휴지통 이동
  - 복구
  - FIFO 영구삭제
- 구체적으로 어느 코드 모듈이 최종 owner인지까지는 원문 미확정이므로 open issue로 남긴다.
- 다만 **값 변경을 발생시킨 처리 로직이 같은 처리 흐름 안에서 재판단/갱신해야 한다**는 점은 구현 방어선으로 고정한다.

### 12-2. 첨부 최종 확인 완료
- 사람의 최종 검토 체크값이다.
- **손가락 사진이 있어야 true 가능**하다.
- 아래 상황이면 자동 false 해제된다.
  - 새 첨부 추가
  - 유형 변경
  - 삭제
  - 휴지통 이동/복구
  - 영구삭제 처리(FIFO 포함)
  - 마지막 손가락 사진 상실

#### 구현 제약
- true는 운영자 판단값이다.
- false reset은 시스템 write-back이 필요하다.
- 이 속성은 formula가 아니므로, 첨부 관련 이벤트 후 자동으로 유지된다고 가정하면 안 된다.
- 다만 false reset의 세부 owner 분해는 원문 미확정이므로 open issue로 남긴다.

### 12-3. 발송 준비 완료(자동)
true 조건:
- `영문 검수 완료 = true`
- `첨부 최종 확인 완료 = true`
- `출력 확인 완료 = true`

즉 위 3개가 모두 true일 때만 true다.

### 12-4. 운영 의미
- `손가락 사진 있음`: 첨부 품질 전제
- `첨부 최종 확인 완료`: 사람 최종 검토 전제
- `발송 준비 완료(자동)`: 발송 가능 여부의 파생 판단

## 14. R2 저장 키 구조와 Notion 첨부 DB 연결 규칙

### 14-1. 저장 키 구조
- 임시 경로: `tmp/{접수번호}/{순번}_{업로드시각}.{확장자}`
- 본경로: `attachments/{접수번호}/{순번}_{업로드시각}.{확장자}`

### 14-2. 저장 규칙
- 순번은 재사용하지 않는다.
- 원본 파일명은 그대로 허용하되, 내부 저장 키는 별도로 생성한다.
- tmp 저장 확인 후 본경로 업로드로 이동한다.
- **첨부 DB `R2 Key`에는 최종 본경로 key만 저장한다.**

### 14-3. 첨부 DB 연결 규칙
첨부 1개 처리 시 순서:
1. 검증
2. 임시 저장 확인
3. 본경로 업로드
4. 첨부 DB 기록
5. 사고건 relation 연결
6. 사고 DB `첨부 업로드 상태` 갱신

### 14-4. 첨부 DB에 최소한 연결되어야 하는 값
- `사고건` relation
- `파일명`
- `R2 Key`
- `첨부 유형`(운영 지정)
- `상태`
- `표시 순서`(운영상 필요)

### 14-5. R2와 Notion의 역할 분리
- 원본 저장소 기준: **R2**
- 메타데이터/분류/삭제 추적 기준: **첨부 DB**
- 사고건 화면 집계/발송 준비 연결 기준: **사고 DB relation 및 파생값**

### 14-6. 휴지통 / FIFO 안전 운영 경계
- 휴지통 첨부는 7일 동안 복구 가능하다.
- 만료 휴지통 정리는 FIFO보다 먼저 적용된다.
- FIFO는 만료 휴지통 정리 후에도 저장소가 5GB를 초과하는 경우에만 적용될 수 있다.
- FIFO 삭제는 휴지통을 거치지 않고 첨부 DB row를 `영구삭제` 상태로 정리한다.
- `check:fifo-trash-candidates` 같은 live-read 검증은 deterministic parity, scenario execution, baseline, CI, product wiring과 분리된 수동 확인 경계로 유지한다.
- 오전 8시 정리 스케줄의 실제 cleanup owner, 5GB 판단에 포함할 R2/storage population, `영구삭제 예정 시각` 계산식은 아직 구현 기준으로 닫지 않는다.

## 15. Queue payload에 반드시 필요한 데이터와 제한사항

### 15-1. 고정 스키마
Queue 메시지는 접수건당 1개이며, 아래 스키마로 잠근다.

```json
{
  "version": 1,
  "receiptNumber": "202604031224-2839",
  "pageId": "notion_page_id",
  "attachmentCount": 2,
  "retryCount": 0,
  "attachments": [
    {
      "seq": 1,
      "tmpKey": "tmp/202604031224-2839/0001_20260403122430.jpg",
      "originalFileName": "finger.jpg",
      "contentType": "image/jpeg",
      "sizeBytes": 1234567
    }
  ]
}
```

### 15-2. 필드 의미
- `version`: payload 버전. 1차 MVP는 `1` 고정
- `receiptNumber`: 로그/R2 경로/운영 추적 공통 키
- `pageId`: 사고 DB 페이지 연결용
- `attachmentCount`: 첨부 개수 검산용
- `retryCount`: 최대 2회 자동 재시도 제어용
- `attachments[].seq`: 동일 사고건 내 표시 순서와 첨부 ID 생성의 기준
- `attachments[].tmpKey`: Consumer가 찾을 임시 객체 key
- `attachments[].originalFileName`: 첨부 DB `파일명` 입력용
- `attachments[].contentType`: 확장자/MIME 검증용
- `attachments[].sizeBytes`: 용량 검산/로그용

### 15-3. 제한사항
- payload는 경량 유지가 원칙이다.
- 첨부 원본 바이트, 본문 전체, 대용량 메타데이터를 payload에 중복 적재하지 않는다.
- `attachments[]`의 각 요소는 위 스키마 외 필수 필드를 추가하지 않는다.
- `첨부 ID`는 Worker가 아니라 Consumer에서 `receiptNumber + seq` 조합으로 생성한다.

### 15-4. 재시도 규칙
- 파일별 최대 2회 자동 재시도
- 그래도 실패하면 관리자 수동 보완 업로드 대상으로 넘긴다.
- 고객이 다시 올리지 않아도 접수는 유지한다.

## 16. 데이터 정합성 위험 포인트

### 16-1. schema drift
- live DB 속성명과 코드 상수명이 다르면 저장 실패 또는 잘못된 필드 매핑이 난다.
- 특히 아래 항목은 옵션명 drift 위험이 높다.
  - `Consent for Promotional Use`
  - `Was There A Visible Injury Mark?`
  - `Type of blade being used`
  - `Were There Other Devices Being Used When the Cut was Made?`
  - `첨부 업로드 상태`
  - `첨부 유형`
  - `영문화 모드`
  - `발송 방식`

### 16-2. 사고 DB와 첨부 DB의 이중 원천 위험
- 첨부 개수를 숫자 필드로 직접 저장하면 rollup과 충돌할 수 있다.
- `손가락 사진 있음`을 운영자가 수동 체크박스로만 유지하면 첨부 DB 실제 상태와 어긋날 수 있다.
- 첨부 상태를 사고 DB와 첨부 DB에 각각 독립 기록하면 drift가 난다.

### 16-3. R2와 첨부 DB 불일치 위험
- R2 업로드는 성공했는데 첨부 DB 기록이 실패하면 고아 객체가 생긴다.
- 첨부 DB는 있는데 R2 원본이 없으면 파일 열기/복구/FIFO 판단이 깨진다.
- FIFO 삭제 후 첨부 DB 상태가 늦게 바뀌면 `손가락 사진 있음` 계산이 틀릴 수 있다.

### 16-4. checkbox write-back 누락 위험
- `손가락 사진 있음`은 formula가 아니라 checkbox다.
- `첨부 최종 확인 완료`도 formula가 아니라 checkbox다.
- 따라서 첨부 추가/삭제/복구/유형 변경/FIFO 후 write-back이 누락되면 발송 준비 판단이 틀어진다.

### 16-5. 상태 전환 위험
- button 동작 상세가 명세에 없으므로, button이 무엇을 함께 갱신하는지 추측 구현하면 위험하다.
- `상태`와 `반려 변경(자동)/진행중 변경(자동)/완료 변경(자동)`의 갱신 주체가 미확정이라 자동화 충돌 여지가 있다.

### 16-6. date/time 정합성 위험
- `Date of Occurence`는 날짜와 시간 결합값인데, “정확한 시간을 잘 모르겠습니다”를 어떻게 저장할지 미확정이다.
- 시간 미상 입력이 들어오면 date 단일값 구성 방식이 달라질 수 있다.

### 16-7. 관리자 업로드 검색 오탐 위험
- 4자리 검색은 연락처 뒷4자리와 접수번호 뒷4자리를 동시에 보므로 동명이건/동일 뒷자리 건이 겹칠 수 있다.
- 최근 20건 제한으로 대상 누락 가능성을 운영상 인지해야 한다.

### 16-8. formula / rollup 조회 오차 위험
- formula/rollup은 파생값이며 직접 수정 불가다.
- Notion API 조회 방식에 따라 formula/rollup/relation 값은 축약 또는 pagination 이슈가 있을 수 있다.
- 구현 시 파생값을 맹신하지 말고 원천 relation/첨부 DB 상태와 함께 확인할 수 있어야 한다.

### 16-9. 후보 속성 오남용 위험
- `mime type`, `file size`, `hash`, `출처` 등 후보 속성을 코드가 전제로 삼으면 live DB에서 실패한다.
- 후보 속성은 문서에 적혀 있어도 아직 실제 schema가 아니다.

### 16-10. 미분류 첨부 처리 방어선
- 원문은 AI 자동분류를 금지하고 병준 수동 분류를 전제로 한다.
- 고객 웹폼 첨부는 첨부 DB 생성 시 `첨부 유형=null`로 저장하고, `첨부 분류 대기` 뷰에서 병준이 수동 분류한다.
- 관리자 보완 업로드는 업로드 시점에 `첨부 유형`을 즉시 지정한다.
- `손가락 사진 있음` 계산에는 `첨부 유형=손가락 사진`인 현재 첨부만 반영한다.

## 19. 한 줄 결론
이 문서의 기준은 **사고건은 사고 DB가, 첨부 메타데이터는 첨부 DB가, 원본 파일은 R2가, 운영 on/off는 운영 설정 DB가 맡고, 접수 성공은 본문 저장 성공으로 판단하며, 첨부 실패·알림 실패·접수증 실패는 접수 실패와 분리한다**는 점이다.
