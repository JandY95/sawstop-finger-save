# WEBFORM_UI_SPEC.md

## 0. 문서 정보

- 문서명: `WEBFORM_UI_SPEC.md`
- 문서 목적: SawStop Finger Save 고객 웹폼의 **화면 구조 / 문구 / 입력 UX / 오류 UX / 완료 화면 / 첨부 UX / 프론트 상태 전이**를 바이브코딩에 바로 사용할 수 있는 수준으로 잠근다.
- 최상위 기준 문서: `sawstop_finger_save_vibe_coding_workflow_spec_final_20260409.md`
- 데이터 기준 문서: `DB_SCHEMA_AND_MAPPING.md`
- 제품 기준 문서: `PRD.md`
- 기술 기준 문서: `TRD.md`
- 구현 분해 기준 문서: `IMPLEMENTATION_BREAKDOWN.md`
- 문서 상태: `v1.1 Locked (2026-04-11 시니어 엔지니어 잠금 결정 반영본)`

### 0-1. 문서 역할
이 문서는 **웹폼 UI 전용 문서**다.

이 문서가 잠그는 것:
1. 웹폼 화면 순서
2. 각 구역의 제목/설명/도움말
3. 필드 라벨과 프론트 입력 유형
4. 프론트 검증 UX
5. 첨부 UX
6. 제출 중/성공/실패 화면 상태
7. 반응형/접근성/가독성 기준

이 문서가 잠그지 않는 것:
- Notion DB 속성명/타입/허용값의 source of truth
- Queue/Consumer 내부 동작
- R2 저장 구현 상세
- 관리자 업로드 페이지 UI
- 영문 리포트 본문 템플릿 상세

### 0-2. 문서 우선순위
충돌 시 우선순위는 아래와 같다.

1. `sawstop_finger_save_vibe_coding_workflow_spec_final_20260409.md`
2. `DB_SCHEMA_AND_MAPPING.md`
3. `PRD.md`
4. `TRD.md`
5. `WEBFORM_UI_SPEC.md`
6. `IMPLEMENTATION_BREAKDOWN.md`

### 0-3. 핵심 원칙
- 웹폼은 **고객이 쉽게 입력하는 화면**이다.
- 저장 구조는 UI가 아니라 `DB_SCHEMA_AND_MAPPING.md`와 `TRD.md`를 따른다.
- 이 문서는 **새 기능 제안서가 아니다.**
- live schema에 없는 속성을 새로 전제하지 않는다.
- 첨부 원본 기준 저장소는 R2이며, UI는 이를 직접 드러내지 않는다.
- 고객 완료 화면에는 내부 상태, page_id, Queue 상태, 관리자용 오류코드를 노출하지 않는다.

## 1. 웹폼의 목표

### 1-1. 고객 관점 목표
- 처음 보는 사용자도 큰 설명 없이 입력할 수 있어야 한다.
- 모바일과 PC에서 모두 읽기 쉬워야 한다.
- 사진이 없어도 접수는 가능해야 한다.
- 접수 후에는 복잡한 내부 상태가 아니라 **접수번호**만 분명히 보이면 된다.

### 1-2. 운영 관점 목표
- 웹폼 필드명이 서버 수신 키와 안정적으로 연결되어야 한다.
- 필수/선택 항목이 명확해야 한다.
- 프론트 검증과 서버 검증이 서로 충돌하지 않아야 한다.
- 첨부 UX가 후행 운영 흐름(분류, 보완 업로드, 최종 검토)을 방해하지 않아야 한다.

### 1-3. 바이브코딩 관점 목표
- `render.ts` 또는 프론트 컴포넌트에서 화면을 한 번에 그릴 수 있어야 한다.
- `constants.ts`, `types.ts`, `normalize.ts`, `validate.ts`, `mapper.ts`와 역할 충돌이 없어야 한다.
- 디자이너식 추상 표현보다 **실제 구현 가능한 라벨/문구/상태명**을 사용한다.

## 2. 다른 문서와의 관계

### 2-1. `DB_SCHEMA_AND_MAPPING.md`와의 관계
- 저장 속성명, 저장 규칙, 허용값, relation 제약은 `DB_SCHEMA_AND_MAPPING.md`가 기준이다.
- 이 문서는 그 데이터를 **고객이 입력하는 화면 형태**로만 정리한다.
- 이 문서는 DB 속성명을 바꾸지 않는다.

### 2-2. `PRD.md`와의 관계
- PRD가 말하는 고객 접수, 완료 화면, 첨부 선택 입력, 후행 첨부 처리 분리 원칙을 화면 수준으로 구체화한다.
- PRD 범위를 넓히는 새 기능을 추가하지 않는다.

### 2-3. `TRD.md`와의 관계
- TRD의 `GET /`, `POST /submit`, normalize, validate, Queue 흐름을 UI 입력 순서와 맞춘다.
- 웹폼 필드명과 서버 계약은 TRD를 따른다.
- 이 문서는 서버 내부 스키마를 직접 정의하지 않는다.

### 2-4. `IMPLEMENTATION_BREAKDOWN.md`와의 관계
- 구현 단계 중 `웹폼 필드 계약 고정`, `render.ts`, `types.ts`, `fixtures` 작업의 화면 기준서로 사용한다.
- 구현 순서를 바꾸지 않는다.

## 3. 화면 구조

### 3-1. 단일 화면 원칙
웹폼은 **한 페이지 구역형**으로 유지한다.

순서는 아래와 같이 고정한다.
1. 연락받으실 정보
2. 사고가 발생한 때와 사람
3. 손가락과 상처 정보
4. 기계 및 카트리지 정보
5. 작업 당시 정보
6. 사고 설명
7. 사진 첨부 및 동의

### 3-2. 상단 고정 요소
#### 제목
`SAWSTOP “Finger Save” 사례 접수`

#### 안내문
`본 양식은 SawStop 사고 보고 및 확인을 위한 접수 양식입니다.`

`제출하신 내용은 내부 검토를 위해 저장되며, 개인정보와 식별 정보는 기밀로 관리됩니다.`

#### 필수 안내
`* 표시는 필수 입력 항목입니다.`

### 3-3. 하단 제출 영역
- 기본 버튼 문구: `접수하기`
- 제출 중 버튼 문구: `접수 중입니다...`
- 제출 버튼은 화면 하단에 1개만 둔다.
- 첨부 안내와 홍보 활용 동의 아래에 둔다.

---

## 4. 화면 레이아웃 기준

### 4-1. 공통 배치
- 데스크톱: 중앙 정렬 1컬럼 기준
- 모바일: 1컬럼 그대로 유지
- 좌우 여백은 충분히 확보한다.
- 각 섹션은 카드형 또는 박스형으로 구분한다.
- 섹션 간 간격은 필드 간 간격보다 더 크게 둔다.

### 4-2. 섹션 카드 규칙
각 섹션은 아래 요소를 기본으로 갖는다.
- 섹션 번호 또는 제목
- 섹션 제목
- 필요 시 보조 설명 1~2줄
- 입력 필드 그룹

### 4-3. 필드 배치 규칙
- 기본은 세로 배치다.
- 모바일 우선으로 설계한다.
- 아래 두 쌍만 데스크톱에서 2열 배치 허용:
  - 연락처 / 이메일
  - 사고 발생일 / 발생 시간
- 나머지는 1열 유지가 안전하다.

### 4-4. 타이포/가독성 원칙
- 라벨은 입력칸 위에 둔다.
- 필수 표시는 라벨 오른쪽에 `*`로 표시한다.
- placeholder는 라벨 대체가 아니라 예시 보조로만 쓴다.
- 도움말 문구는 입력칸 아래 작은 글씨로 둔다.
- 오류 문구는 도움말보다 아래, 입력칸 바로 아래 한 줄로 둔다.

## 5. 필드 상세 명세

## 5-1. 연락받으실 정보

### 1) 회사 또는 학교명
- 라벨: `회사 또는 학교명`
- 입력 유형: text
- 필수 여부: 선택
- placeholder: `예: 동아대학교 / OO목공방`
- 저장 연결: `Business or School Name (NA if Not Applicable)`
- UX 메모: 미입력 가능. 비어 있어도 접수 가능.

### 2) 연락처
- 라벨: `연락처 *`
- 입력 유형: tel
- placeholder: `010-1234-5678`
- 저장 연결: `Phone`
- UX 메모:
  - 숫자 입력 중심
  - 자동 하이픈 적용
  - 모바일 숫자 키패드 우선

### 3) 이메일
- 라벨: `이메일 *`
- 입력 유형: email
- placeholder: `name@example.com`
- 저장 연결: `Email`
- UX 메모:
  - 입력 중 공백 제거
  - 저장 전 소문자 정리
  - 모바일 이메일 키보드 우선

## 5-2. 사고가 발생한 때와 사람

### 4) 사고 발생일
- 라벨: `사고 발생일 *`
- 입력 유형: date
- 저장 연결: `Date of Occurence`

### 5) 발생 시간
- 라벨: `발생 시간 *`
- 입력 유형: time + 보완 옵션
- 저장 연결: `Date of Occurence`
- UI 규칙:
  - 기본은 시간 선택 입력
  - 아래에 체크 옵션 제공: `정확한 시간을 잘 모르겠습니다`
  - 체크 시 시간 입력 비활성화
  - 보조 문구: `정확하지 않으면 대략적인 시간대만 입력하셔도 됩니다.`

### 6) 작업자 이름
- 라벨: `작업자 이름`
- 입력 유형: text
- 저장 연결: `Operator Name`

### 7) 접촉된 사람(피해자)
- 라벨: `접촉된 사람(피해자)`
- 입력 유형: text
- 저장 연결: `Name of Person Who Touched the Blade`

## 5-3. 손가락과 상처 정보

### 8) 어느 부위가 톱날에 닿았나요?
- 라벨: `어느 부위가 톱날에 닿았나요? *`
- 입력 유형: text
- placeholder: `예: 오른손 검지 / 왼손 엄지`
- 저장 연결: `Body Part Contacted (right or left hand, finger, thumb, etc.)`

### 9) 상처가 보였나요?
- 라벨: `상처가 보였나요? *`
- 입력 유형: select
- 선택값:
  - `예 (YES)`
  - `아니요 (NO)`
- 저장 연결: `Was There A Visible Injury Mark?`

### 10) 상처 치료 방법
- 라벨: `상처 치료 방법`
- 입력 유형: text
- placeholder: `예: 소독 / 밴드 / 병원 방문`
- 저장 연결: `Wound treatment methods`

### 11) 쏘스탑이 없었다면 상처가 어느 정도였을 것 같나요?
- 라벨: `쏘스탑이 없었다면 상처가 어느 정도였을 것 같나요?`
- 입력 유형: textarea 또는 긴 text input
- 저장 연결: `Estimate of the injury if it were to have occured while using a non-SawStop saw`

## 5-4. 기계 및 카트리지 정보

### 12) 기계 시리얼 넘버
- 라벨: `기계 시리얼 넘버 *`
- 입력 유형: text
- placeholder: `예: C123456789`
- 저장 연결: `Saw Serial Number`
- 안내 문구:
  - `영문 C, P, I 중 하나로 시작하는 시리얼 번호를 입력해 주세요.`
  - `예: C123456789 / P123456789 / I123456789`

### 13) 브레이크 카트리지 시리얼 넘버
- 라벨: `브레이크 카트리지 시리얼 넘버`
- 입력 유형: text
- placeholder: `사진으로 나중에 보완하셔도 됩니다.`
- 저장 연결: `Brake Cartridge Serial Number`

### 14) 장착 날(블레이드) 종류
- 라벨: `장착 날(블레이드) 종류`
- 입력 유형: select
- 선택값:
  - `10" Standard`
  - `8" Dado`
- 저장 연결: `Type of blade being used`

### 15) 날 세부
- 라벨: `날 세부`
- 입력 유형: text
- placeholder: `예: 40T, kerf 3.2mm`
- 저장 연결: `Saw Blade Details`

## 5-5. 작업 당시 정보

### 16) 작업하신 재료의 재질은 무엇인가요?
- 라벨: `작업하신 재료의 재질은 무엇인가요? *`
- 입력 유형: text
- placeholder: `예: 원목 / 합판 / MDF`
- 저장 연결: `Type of Material Being Cut?`
- 주의: UI는 자유입력이고 저장도 text다.

### 17) 사고 당시 작업물 크기와 작업 방식은 어떻게 되었나요?
- 라벨: `사고 당시 작업물 크기와 작업 방식은 어떻게 되었나요?`
- 입력 유형: text
- placeholder: `예: 폭 100mm 정도의 원목을 길이 방향으로 절단`
- 저장 연결: `Workpiece Size & Cut Type`

### 18) 사고 당시 톱날 안전장치가 장착되어 있었나요?
- 라벨: `사고 당시 톱날 안전장치가 장착되어 있었나요?`
- 입력 유형: text
- placeholder: `예: 블레이드 가드 사용 / 라이빙나이프만 장착 / 없음`
- 저장 연결: `Was a Blade Guard, Riving Knife or Splitter in Place? (please specify which, if any)`

### 19) 사고 당시 사용한 보조장치가 있었나요?
- 라벨: `사고 당시 사용한 보조장치가 있었나요?`
- 입력 유형: multi_select
- 선택값:
  - `푸시스틱 (Push Stick)`
  - `페더보드 (Feather Board)`
  - `마이터 게이지 (Miter Gauge)`
  - `기타 보조장치 (Other)`
  - `사용하지 않음 (None)`
- 저장 연결: `Were There Other Devices Being Used When the Cut was Made?`
- UX 메모:
  - `사용하지 않음 (None)`은 다른 항목과 동시 선택 불가

### 20) 사고 당시 장갑을 착용하고 있었나요?
- 라벨: `사고 당시 장갑을 착용하고 있었나요?`
- 입력 유형: select
- 선택값:
  - `예 (YES)`
  - `아니요 (NO)`
- 저장 연결: `Was the saw operator wearing gloves at the time?`

### 21) 사고 당시 재료를 얼마나 빠르게 밀고 있었나요?
- 라벨: `사고 당시 재료를 얼마나 빠르게 밀고 있었나요?`
- 입력 유형: select
- 선택값:
  - `아주 느림 (Very slow)`
  - `느림 (Slow)`
  - `보통 (Normal)`
  - `빠름 (Fast)`
  - `매우 빠름 (Very fast)`
- 저장 연결: `What was the approximate feed rate of the material when the accident occured (inches per second)?`

## 5-6. 사고 설명

### 22) 사고 원인은 무엇이라고 생각하시나요?
- 라벨: `사고 원인은 무엇이라고 생각하시나요?`
- 입력 유형: text
- placeholder: `예: 재료가 밀리면서 손이 함께 들어간 것 같습니다.`
- 저장 연결: `Cause of the Incident (Customer Feedback)`

### 23) 사고 당시 상황을 적어 주세요.
- 라벨: `사고 당시 상황을 적어 주세요. *`
- 입력 유형: textarea
- placeholder: `예: 재료를 밀면서 작업하던 중 재료가 흔들렸고, 손이 앞으로 나가 톱날에 닿았습니다. 이후 기계가 바로 멈췄습니다.`
- 저장 연결: `To the best of your ability, please describe the circumstances of how the accident happened`
- UX 메모:
  - 최소 4줄 높이 권장
  - 모바일에서도 줄바꿈 입력이 쉬워야 함

## 5-7. 사진 첨부 및 동의

### 24) 사진을 첨부하시겠어요?
- 라벨: `사진을 첨부하시겠어요?`
- 입력 유형: file (multiple)
- 필수 여부: 선택
- 저장 연결: `R2 + SAWSTOP 첨부 관리 DB + 사고 DB relation`
- 안내 문구:
  - `손가락 사진이나 브레이크 카트리지 사진이 있으시면 첨부해 주세요.`
  - `사진이 없어도 접수는 가능합니다.`
  - `최대 4장까지, 각 파일은 10MB 이하로 업로드해 주세요.`
  - `이미지 파일만 업로드할 수 있습니다. 허용 형식: jpg, jpeg, png, webp, heic, heif`
  - `브레이크 카트리지 사진은 시리얼 번호가 보이도록 촬영해 주시면 확인에 도움이 됩니다.`

### 25) 홍보 활용 동의
- 라벨: `SawStop이 고객님의 사용 후기 또는 “Finger Save” 사례를 외부 홍보 또는 마케팅 자료로 활용하는 데 동의해 주시겠습니까? *`
- 입력 유형: select
- 선택값:
  - `동의 (YES)`
  - `미동의 (NO)`
- 저장 연결: `Consent for Promotional Use`
- 보조 문구:
  - `개인정보와 식별 정보는 기밀로 관리되며, 별도 동의가 없는 한 SawStop 내부 용도로만 사용됩니다.`

## 6. 첨부 UI 상세 규칙

### 6-1. 업로드 영역
- 드래그 앤 드롭 영역 자체를 클릭 가능하게 만든다.
- 모바일에서는 카메라/앨범 선택이 가능해야 한다.
- 파일 선택 전 기본 상태에서 `0/4`를 표시한다.
- 파일이 추가되면 `1/4`, `2/4`처럼 즉시 갱신한다.

### 6-2. 첨부 리스트
- 선택된 파일은 썸네일 목록으로 보여준다.
- 각 썸네일마다 개별 삭제 버튼을 둔다.
- 파일 선택 순서를 유지한다.
- 파일명은 너무 길면 말줄임 처리하되 hover/tap 시 전체 확인 가능하게 한다.

### 6-3. 프론트 파일 처리 규칙
- EXIF 방향 보정을 적용한다.
- 동일 파일 중복 선택 시 해시 기준 중복 제거를 시도한다.
- 형식/용량 오류는 해당 파일만 제외한다.
- 일부 파일만 제외되어도 나머지 정상 파일은 유지한다.

### 6-4. 저장/분류 관련 UI 원칙
- 고객 웹폼에서는 **첨부 유형을 직접 고르게 하지 않는다.**
- 고객 화면은 첨부 분류 UI를 가지지 않는다.
- 분류는 운영 검토 또는 관리자 보완 업로드 단계에서 처리한다.
- 따라서 웹폼 첨부는 UI 상 `미분류 접수 첨부`로 보는 전제가 안전하다.

### 6-5. 첨부 관련 비노출 원칙
고객 화면에 아래는 보여주지 않는다.
- R2 Key
- tmp 경로
- 첨부 DB row 정보
- Queue 처리 상태
- `첨부 업로드 상태`
- `손가락 사진 있음`

## 11. open issue를 UI 기준으로 잠근 확정 결정

이 섹션은 **2026-04-11 기준으로 실제 구현 잠금본에 반영한 확정 결정**이다.
UI 문서는 화면/입력 UX만 다루되, 아래 항목은 관련 하위 문서와 정합성이 맞도록 함께 잠갔다.

### OI-04. 사고 DB `첨부(선택)` file 속성의 역할
확정 결정:
- 고객 웹폼 UI에서는 이 속성을 노출하지 않는다.
- 구현 기준 저장소는 R2다.
- 1차 MVP에서 사고 DB `첨부(선택)`은 **원본 저장소가 아니라 보조 표시용/비사용 영역**으로 잠근다.
- 즉, 웹폼/Consumer 로직은 `첨부 DB relation + R2`를 기준으로 구현한다.

### OI-05. `Date of Occurence` 시간 미상 처리
확정 결정:
- UI에는 `정확한 시간을 잘 모르겠습니다` 체크를 둔다.
- 체크 시 `occurredDate`는 필수 유지, `occurredTime`은 비활성화한다.
- 서버 저장 규칙은 **선택한 날짜의 정오 `12:00 (Asia/Seoul)`** 로 잠근다.
- 완료 화면이나 고객 문구에는 이 내부 저장 규칙을 노출하지 않는다.

### OI-07. `첨부 ID` 생성 규칙
확정 결정:
- `ATT-{receiptNumber}-{seq4}` 형식으로 잠근다.
- 예: `ATT-202604031224-2839-0001`
- 동일 접수 내 순번은 재사용하지 않는다.

### OI-08. Queue payload의 첨부별 참조 정보 shape
확정 결정:
- 경량 payload 유지 원칙을 지킨다.
- 첨부별 참조 정보는 아래로 잠근다.
```json
{
  "version": 1,
  "receiptNumber": "202604031224-2839",
  "pageId": "notion_page_id",
  "attachmentCount": 2,
  "retryCount": 0,
  "attachments": [
    {
      "seq": 1,
      "tmpKey": "tmp/202604031224-2839/0001_20260403122430.jpg",
      "originalFileName": "finger.jpg",
      "contentType": "image/jpeg",
      "sizeBytes": 1234567
    }
  ]
}
```
- 첨부 원본 바이트 자체를 payload에 넣지 않는다.

### OI-09. `표시 순서` 산정 규칙
확정 결정:
- 고객 웹폼에서 선택한 순서를 그대로 1, 2, 3... 부여한다.
- 관리자 보완 업로드는 기존 최대값 다음 순번부터 이어붙인다.
- 삭제/휴지통 이동/복구/FIFO 후에도 기존 순번은 재사용하지 않는다.

### OI-14. `미리보기 링크` / `썸네일` 생성 방식
확정 결정:
- 1차 MVP에서는 아래처럼 단순화한다.
  - `미리보기 링크`: R2 공개/서명 URL 또는 Worker 프록시 URL
  - `썸네일`: 별도 썸네일 생성 전까지는 `미리보기 링크`와 동일 URL 사용
- 즉, 1차는 **썸네일 전용 이미지 변환 파이프라인 없이도 구현 가능**하게 잠근다.

### OI-19. 고객 웹폼 첨부의 분류 전 저장 방식
확정 결정:
- 고객 웹폼 첨부는 업로드 시점에 첨부 유형을 묻지 않는다.
- 고객 웹폼 첨부는 첨부 DB 저장 시 `첨부 유형=null`로 생성한다.
- 운영자는 `첨부 분류 대기` 뷰에서 수동 지정한다.
- 관리자 보완 업로드는 업로드 시점에 첨부 유형을 즉시 선택한다.

## 15. 한 줄 결론

`WEBFORM_UI_SPEC.md`는 데이터 문서를 건드리지 않고도 **웹폼 화면과 UX를 잠가서 바이브코딩 품질을 올리는 보조 문서**다. 이 문서가 있으면 `render.ts`와 프론트 구현이 훨씬 덜 흔들리고, PRD/TRD/DB 문서를 UI 세부사항으로 더럽히지 않아도 된다.
