## 1. 제품 개요

SawStop Finger Save 프로젝트는 국내 SawStop 사용자 사고 정보를 **한국어 웹폼**으로 쉽게 접수받고, 이를 **Notion 사고 페이지** 중심으로 관리하며, 같은 페이지 본문에서 **영문 리포트**를 작성·수정하고, 최종적으로 **SawStop 본사 전달**까지 이어지는 운영 구조를 만드는 제품이다.

이 제품의 1차 목표는 고객이 어렵지 않게 사고를 접수하고, 운영자가 Notion을 중심으로 사고건을 복원·검토·보완·전달할 수 있게 만드는 것이다.

핵심 구조는 아래와 같다.

- 고객 입력 채널: 별도 한국어 웹폼
- 저장/처리: Cloudflare Workers
- 첨부 원본 저장소: Cloudflare R2
- 작업판: Notion (`SAWSTOP 사고 보고`, `SAWSTOP 첨부 관리`, `SAWSTOP 운영 설정`)
- 영문 리포트: 같은 사고 페이지 본문
- 발송 운영: 1차는 수동, 이후 조건 충족 시 반자동 확장 가능

## 2. 문제 정의

현재 SawStop 사고 접수와 후속 정리는 고객 입력, 첨부 관리, 영문 정리, 출력 확인, 본사 전달이 하나의 일관된 흐름으로 잠겨 있지 않으면 운영 복원성이 떨어지고, 누락과 재작업이 발생하기 쉽다.

이 프로젝트가 해결하려는 핵심 문제는 다음과 같다.

### 2-1. 고객 접수 문제
- 고객이 사고 정보를 쉽게 제출할 수 있는 **한국어 중심 접수 경로**가 필요하다.
- 사진 첨부가 중요하지만, 접수 단계에서 강제하면 접수 장벽이 높아진다.
- 고객은 복잡한 내부 상태보다 **접수번호 확인**만 바로 되면 된다.

### 2-2. 운영 복원 문제
- 사고 원문, 첨부, 영문 리포트, 출력 확인, 발송 준비 상태가 한 흐름으로 복원되어야 한다.
- 첨부 처리 실패가 전체 접수를 망가뜨리면 운영 복구가 어려워진다.
- 손가락 사진, 카트리지 사진, 시리얼 확인 등 후속 보완 흐름이 별도로 존재해야 한다.
- 영문 초안 작성 후에도 첨부 보완이 가능해야 하며, 다시 같은 사고 페이지로 돌아와 작업을 이어갈 수 있어야 한다.

### 2-3. 데이터 정합성 문제
- live Notion DB 기준과 구현 코드가 어긋나면 저장 실패나 drift가 발생한다.
- relation / rollup / formula / button은 역할이 다르므로, 파생값과 원천값을 섞어 쓰면 정합성이 깨진다.
- 첨부 원본, 첨부 메타데이터, 사고건 파생 상태를 각각 올바른 저장소에 분리해야 한다.

### 2-4. 운영 예외 처리 문제
- 고객 제출 실패 시 사용자에게는 단순하고 안전한 메시지만 보여야 한다.
- 내부 오류 추적은 고객 화면이 아니라 서버 로그와 관리자 알림 기준으로 처리되어야 한다.
- 휴지통, 복구, 만료 삭제, FIFO 같은 첨부 수명주기까지 포함해 상태가 일관되게 유지되어야 한다.

## 3. 목표

### 3-1. 제품 목표
- 고객이 **한 페이지형 한국어 웹폼**으로 사고를 접수할 수 있게 한다.
- 접수 즉시 **접수번호**를 생성하고, 사고건을 Notion에 저장한다.
- 첨부 원본은 R2에 저장하고, 첨부 메타데이터는 별도 첨부 DB로 관리한다.
- 운영자가 같은 사고 페이지에서 원문과 영문 리포트를 함께 보고 수정할 수 있게 한다.
- 필요 시 **관리자 수동 보완 업로드**로 첨부를 보강할 수 있게 한다.
- 발송 전 필요한 확인 상태를 분리해 관리하고, **본사 전달까지**를 1차 완료선으로 삼는다.

### 3-2. 운영 목표
- 접수 성공 기준을 명확히 고정한다: **사고 DB 페이지 생성 + 매핑 대상 속성 저장 + 기본 본문 저장이 모두 성공해야 접수 성공**
- 첨부 실패, 관리자 알림 실패, 고객 접수증 메일 실패를 접수 실패와 분리한다.
- 손가락 사진은 접수 단계 선택으로 두되, **최종 리포트 전 필수 확보** 원칙을 유지한다.
- live Notion DB 제약과 충돌하지 않는 범위에서만 제품을 정의한다.
- 첨부 변경 이벤트가 생기면 관련 검토 상태가 다시 반영되도록 한다.

### 3-3. 문서 목표
- TRD와 실제 구현 단계 분해의 기준이 되는 요구사항 문서 역할을 한다.
- 운영 목적, 사용자, 흐름, 범위, 성공 기준, 제약사항을 분리해 해석 충돌을 줄인다.
- 데이터 상세 문서를 반복하지 않되, 제품 요구사항이 데이터/운영 제약과 충돌하지 않게 유지한다.

## 4. 사용자와 역할

### 4-1. 국내 SawStop 고객
- 한국어 웹폼에 사고 정보를 입력한다.
- 필요 시 사진을 첨부한다.
- 접수 후 접수번호를 확인한다.
- 후속 요청이 있을 경우 손가락 사진 또는 카트리지 사진을 추가 제출한다.
- 활성화된 브레이크 카트리지를 발송한다.

### 4-2. 운영자(병준)
- 사고건 1차 검토를 수행한다.
- 입력값 누락, 손가락 사진 확보 여부, 첨부 상태, 카트리지 시리얼 확인 가능 여부를 판단한다.
- 필요 시 관리자 보완 업로드 페이지에서 첨부를 추가·교체한다.
- 같은 사고 페이지 본문에서 영문 리포트를 작성·수정한다.
- 출력본과 첨부를 최종 확인하고 본사에 수동 발송한다.
- 발송 실패나 첨부 예외 상황을 운영 메모와 로그 기준으로 추적한다.

### 4-3. SawStop 본사 검토 담당자
- 전달된 영문 리포트와 첨부를 검토한다.
- 후속 카트리지 데이터를 검토한다.
- 최종 결과를 회신한다.

## 5. 핵심 사용자 흐름

### 5-1. 고객 접수 흐름
1. 고객이 공개 웹폼에 사고 정보를 입력한다.
2. Workers가 입력값을 검증하고 접수번호를 생성한다.
3. Workers가 사고 DB 페이지를 생성하고, 매핑 대상 속성과 기본 본문을 저장한다.
4. **사고 DB 페이지 생성 + 매핑 대상 속성 저장 + 기본 본문 저장이 모두 성공해야 접수가 성공한다.**
5. 고객은 완료 화면에서 접수번호를 확인한다.
6. 고객 완료 화면에는 내부 첨부 처리 상태나 내부 오류코드를 기본 노출하지 않는다.
7. 첨부가 있으면 후행 비동기로 R2 저장과 첨부 DB 기록이 진행된다.

### 5-2. 운영자 검토 흐름
1. 운영자가 Notion 사고건을 연다.
2. 원문 입력값, 첨부 존재 여부, 손가락 사진 확보 여부, 시리얼 확인 가능 여부를 확인한다.
3. 필요한 경우 관리자 수동 보완 업로드로 첨부를 추가하거나 교체한다.
4. 같은 사고 페이지 본문에서 영문 리포트 초안을 작성한다.
5. 첨부가 부족하면 보완 업로드 흐름으로 다시 진입한다.
6. 보완 후 같은 사고 페이지로 돌아와 영문 리포트를 계속 수정한다.
7. 영문 리포트, 첨부, 출력본을 확인한다.
8. 발송 준비 완료 조건이 충족되면 본사에 수동 발송한다.

### 5-3. 첨부 수명주기 흐름
1. 첨부는 현재 사고건에 연결된 첨부 메타데이터와 R2 원본으로 관리된다.
2. 삭제는 휴지통 이동을 기본으로 하며, 개별 첨부 단위로 처리한다.
3. 휴지통 첨부는 복구 가능 기간이 지나면 정리 대상이 된다.
4. 저장소 용량 정책이 작동하면 FIFO 삭제가 적용될 수 있다.
5. 첨부 변경이 발생하면 `손가락 사진 있음`, `첨부 최종 확인 완료`, 발송 준비 관련 상태가 다시 반영된다.

### 5-4. 완료선
- 1차 제품 완료선은 **본사 전달까지**다.
- 카트리지 데이터 수집, 본사 분석, 최종 결과 회신은 후속 운영 단계다.

## 6. MVP 범위

MVP는 아래 범위까지만 포함한다.

### 6-1. 고객 접수
- 공개 한국어 웹폼 1개
- 필수/선택 항목 구분
- Turnstile 포함 제출
- 프론트 1차 검증 + 서버 최종 검증
- 완료 화면과 접수번호 반환
- 고객 완료 화면에 내부 첨부 상태 및 내부 오류코드 비노출

### 6-2. 사고건 저장
- 접수번호 생성
- 사고 DB 페이지 생성
- 매핑 대상 속성 저장
- 기본 본문 저장
- 접수 성공 기준 고정: **사고 DB 페이지 생성 + 매핑 대상 속성 저장 + 기본 본문 저장이 모두 성공해야 접수 성공**
- 상태 초기값 반영
- `Date of Occurence`는 날짜와 시간을 1개 date로 저장하며, `정확한 시간을 잘 모르겠습니다` 선택 시 **선택 날짜의 12:00 (Asia/Seoul)** 로 저장한다.

### 6-3. 첨부 처리
- 첨부 최대 4장, 파일당 10MB 이하, 이미지 형식 제한
- 첨부 원본 R2 저장
- 첨부 DB 1행당 1첨부 기록
- 사고건 relation 연결
- `첨부 업로드 상태` 갱신
- 첨부 실패와 접수 실패 분리
- 사고 DB `첨부(선택)` file 속성은 1차 MVP의 원본 저장소나 자동 write 대상이 아니다.
- 첨부 DB `첨부 ID`는 `ATT-{receiptNumber}-{seq4}` 형식으로 생성한다.
- 첨부 `표시 순서`는 고객 웹폼 선택 순서 그대로 사용하고, 관리자 보완 업로드는 기존 최대값 다음 순번부터 이어 붙인다.
- `미리보기 링크`는 R2 접근 URL을 사용하고, `썸네일`은 1차 MVP에서 같은 URL 재사용을 허용한다.

### 6-4. 운영 검토
- 운영자가 Notion 사고 페이지에서 사고건을 검토
- 손가락 사진 확보 여부 판단
- 첨부 유형 수동 분류
- 고객 웹폼 첨부는 접수 시점에 유형을 묻지 않으며, 첨부 DB 생성 시 `첨부 유형`은 빈값(null)으로 저장한 뒤 `첨부 분류 대기` 뷰에서 수동 분류한다.
- 관리자 수동 보완 업로드 페이지 제공
- 관리자 보완 업로드는 업로드 시점에 첨부 유형을 즉시 선택한다.
- 영문 초안 작성 이후에도 보완 업로드 재진입 가능

### 6-5. 영문 리포트 및 출력 확인
- 같은 사고 페이지 본문에서 영문 리포트 작성/수정
- 외부 제목과 기본 템플릿 구조 유지
- 같은 영문 본문을 기준으로 한 최소 웹뷰/PDF 출력 경로 제공
- `출력 확인 완료`를 위한 최소 확인 흐름 제공

### 6-6. 발송 준비 및 본사 전달
- `영문 검수 완료`, `출력 확인 완료`, `첨부 최종 확인 완료` 기반의 발송 준비 판단
- 본사 수동 발송
- 발송 실패 시 메모 기록 가능

### 6-7. 최소 오류 처리
- 고객 실패 시 일반 실패 문구 제공
- 관리자 추적은 서버/호스팅 로그 + 관리자 알림 기준
- 오류코드는 고객 화면에 직접 노출하지 않음

### 6-8. 첨부 상태 동기화의 최소 범위
- `손가락 사진 있음` 재판단
- `첨부 최종 확인 완료` 자동 false 해제
- 휴지통 / 복구 / 만료 삭제 / FIFO에 따른 관련 상태 재반영

## 7. 핵심 기능

### 7-1. 한국어 웹폼 접수
- 고객은 큰 설명 없이 PC/모바일에서 입력 가능해야 한다.
- 웹폼은 한 페이지 구역형 구조를 유지한다.
- 필수 항목만 최소화하고, 사진 첨부는 선택 입력으로 둔다.

### 7-2. 접수번호 생성 및 완료 화면
- 접수번호는 `YYYYMMDDHHmm-연락처뒷4자리` 형식으로 생성한다.
- 고객 완료 화면에는 내부 상태 대신 접수 완료 문구와 접수번호만 표시한다.
- 첨부 상태 요약은 고객 완료 화면의 기본 노출 대상이 아니다.

### 7-3. 사고 DB 저장 우선 처리
- **사고 DB 페이지 생성 + 매핑 대상 속성 저장 + 기본 본문 저장이 모두 성공해야 접수 성공**이다.
- 접수 처리에는 사고 DB 페이지 생성, 매핑 대상 속성 저장, 기본 본문 저장이 포함된다.
- 첨부, 관리자 알림, 고객 접수증 메일은 후행 요소이며 접수 성공과 분리한다.

### 7-4. 첨부 비동기 처리
- 첨부는 Queue/Consumer 기반으로 순차 처리한다.
- 원본 저장소는 R2를 기준으로 사용한다.
- 첨부 메타데이터는 `SAWSTOP 첨부 관리` DB에 저장한다.
- 사고건에는 relation과 파생 상태만 반영한다.

### 7-5. 관리자 수동 보완 업로드
- 공개 웹폼과 분리된 별도 관리자 업로드 페이지를 제공한다.
- 접수번호 자동 포함 링크 지원, 비밀번호 + Turnstile + session cookie 인증, 5회 실패 시 10분 잠금 규칙을 따른다.
- 검색은 완료건을 제외하고 접수번호/연락처 기준으로 사고건을 찾을 수 있어야 한다.
- 결과는 성공 / 부분 성공 / 실패 / 인증 실패를 구분할 수 있어야 한다.

### 7-6. 손가락 사진 확보 및 첨부 검토 상태 관리
- 손가락 사진은 접수 시점에 강제하지 않는다.
- 그러나 최종 리포트 전에는 반드시 확보해야 한다.
- `손가락 사진 있음`은 현재 첨부 상태를 기준으로 재판단되는 운영 체크값이다.
- `첨부 최종 확인 완료`는 사람이 체크하되, 새 첨부 추가, 유형 변경, 삭제, 휴지통 이동/복구, FIFO, 마지막 손가락 사진 상실 시 자동 false 해제가 필요하다.

### 7-7. 같은 사고 페이지 본문에서 영문 리포트 작성
- 별도 영문 리포트 DB를 만들지 않는다.
- 사고건 원문과 영문 리포트를 한 페이지 흐름에서 함께 관리한다.
- 영문 템플릿 항목명은 Notion의 영문 속성명을 기준으로 사용한다.
- 외부 제목은 `Report a Save (Known or Suspected Finger Contact)`을 사용한다.
- `Phone`, `Email`, `Consent for Promotional Use`는 고정 포함 항목으로 본다.

### 7-8. 웹뷰 / PDF 출력 확인
- 웹뷰와 PDF는 같은 영문 리포트 원본 본문을 기준으로 한다.
- 출력은 발송 전 확인 흐름의 일부다.
- 1차 범위에서는 최소 출력 경로와 확인 가능성 확보를 우선하며, 스타일 고도화는 후순위로 둔다.

### 7-9. 발송 준비 상태 관리
- 발송 준비는 수동 메모가 아니라 고정된 체크 흐름으로 판단한다.
- `영문 검수 완료`, `출력 확인 완료`, `첨부 최종 확인 완료`가 모두 true일 때 `발송 준비 완료(자동)`이 true가 된다.
- 1차 발송 방식은 수동을 기본으로 한다.
- 별도 발송 상태 DB는 만들지 않는다.

### 7-10. 최소 오류 처리 및 운영 추적
- 고객 화면에는 일반 실패 문구만 노출한다.
- 고객에게 내부 추적용 오류코드는 노출하지 않는다.
- 운영자는 서버/호스팅 로그와 관리자 오류 알림을 기준으로 문제를 추적한다.

## 9. 성공 기준

### 9-1. 제품 성공 기준
- 고객이 한국어 웹폼으로 사고를 접수할 수 있다.
- **사고 DB 페이지 생성 + 매핑 대상 속성 저장 + 기본 본문 저장이 모두 성공해야 접수가 성공으로 처리된다.**
- 고객은 완료 화면에서 접수번호를 확인할 수 있다.
- 고객 완료 화면에는 내부 첨부 상태와 내부 오류코드가 기본 노출되지 않는다.
- 첨부가 있으면 후행 처리 결과가 `첨부 업로드 상태`에 반영된다.
- 운영자는 같은 사고 페이지에서 원문, 첨부, 영문 리포트, 출력 확인, 발송 준비 상태를 확인할 수 있다.
- 필요 시 관리자 수동 보완 업로드로 첨부를 추가·교체할 수 있다.
- 발송 준비 조건을 확인한 뒤 본사 전달까지 수행할 수 있다.

### 9-2. 운영 성공 기준
- 첨부 일부 실패나 전부 실패가 접수 실패로 처리되지 않는다.
- 손가락 사진이 없더라도 접수는 가능하지만, 최종 리포트 전에는 확보 여부를 명확히 판단할 수 있다.
- 첨부 변경 이벤트 발생 시 검토 상태가 다시 반영된다.
- 휴지통 / 복구 / FIFO 이후에도 관련 상태가 운영 기준에 맞게 유지된다.
- live Notion DB 속성 및 허용값과 충돌 없이 저장된다.

### 9-3. 데이터 성공 기준
- 사고 원문은 사고 DB가, 첨부 메타데이터는 첨부 DB가, 원본 파일은 R2가 담당한다.
- 파생값은 relation / rollup / formula / checkbox write-back 역할을 구분해 처리한다.
- 후보 속성은 실제 구현 전제로 사용하지 않는다.
- formula와 rollup은 직접 쓰기 대상이 아니다.

## 10. 운영상 중요한 제약사항

### 10-1. 접수 성공 기준 고정
- **사고 DB 페이지 생성 + 매핑 대상 속성 저장 + 기본 본문 저장이 모두 성공해야 접수 성공**이다.
- 첨부 실패, 관리자 알림 실패, 고객 접수증 메일 실패는 접수 실패가 아니다.

### 10-2. 손가락 사진 정책
- 접수 단계에서는 선택 입력이다.
- 최종 리포트 전에는 반드시 확보해야 한다.
- 마지막 손가락 사진이 사라지는 경우 관련 검토 상태와 발송 준비 상태는 다시 반영되어야 한다.

### 10-3. 발송 운영 정책
- 1차 발송은 수동 기본이다.
- 반자동은 SMTP 세팅 완료 후에만 확장한다.
- 발송 준비 판단은 별도 DB가 아니라 기존 사고 DB 속성 조합을 사용한다.

### 10-4. 첨부 저장 정책
- 원본 저장소 기준은 R2다.
- Notion file 속성을 원본 저장소로 승격하지 않는다.
- 첨부 1개는 첨부 DB 1행으로 관리한다.

### 10-5. 첨부 lifecycle 정책
- 삭제는 개별 첨부 단위로 처리한다.
- 휴지통 복구 가능 기간과 만료 삭제 운영이 존재한다.
- 저장소 정리 과정에서 FIFO 삭제가 적용될 수 있다.
- 휴지통 / 복구 / 만료 삭제 / FIFO는 첨부 관련 상태 재반영의 트리거로 본다.

### 10-6. live schema 우선 원칙
- live Notion DB 실제 속성이 우선이다.
- 후보 속성은 문서상 후보로만 유지하고 구현 전제로 삼지 않는다.
- 문서와 코드가 다르면 먼저 schema/문서/매핑을 재대조한다.

### 10-7. 파생값 처리 원칙
- relation / rollup / formula / button은 원천 데이터와 동일하게 취급하지 않는다.
- rollup / formula 값은 직접 쓰지 않는다.
- button 존재를 API 자동화 전제로 삼지 않는다.

### 10-8. 오류 처리 원칙
- 고객 실패 문구는 일반 문구를 사용한다.
- 관리자 추적용 오류코드는 고객 화면에 노출하지 않는다.
- 1차 오류 추적은 서버/호스팅 로그와 관리자 알림 기준으로 처리한다.

## 11. 데이터/운영 전제 요약

### 11-1. 데이터 구조 전제
- 핵심 DB는 `SAWSTOP 사고 보고`, `SAWSTOP 첨부 관리`, `SAWSTOP 운영 설정` 3개다.
- 사고건 1건은 사고 DB 페이지 1개다.
- 첨부 1개는 첨부 DB 1행이다.
- 운영 설정은 기본 설정 1행만 사용한다.

### 11-2. 저장 책임 분리
- 고객 원문 사고 정보: 사고 DB
- 첨부 메타데이터 및 분류/삭제 상태: 첨부 DB
- 첨부 원본 파일: R2
- 운영 on/off와 기본 메일 정보: 운영 설정 DB

### 11-3. live schema / 후보 속성 전제
- 사고 DB와 운영 설정 DB는 live 기준 속성만 사용한다.
- 첨부 DB의 `출처`, `최근 변경 이력`, `mime type`, `file size`, `hash`, `접수번호 복제 속성`은 후보 속성이므로 실제 구현 전제로 사용하지 않는다.

### 11-4. relation / rollup / formula / button 제약 요약
- 코드의 relation 쓰기 경로는 첨부 DB `사고건` relation 기준으로 고정한다.
- 사고 DB `첨부 목록`은 읽기 및 집계 기준으로 본다.
- `현재 첨부 개수`, `휴지통 첨부 개수`, `전체 첨부 개수`는 rollup 파생값이다.
- `발송 준비 완료(자동)`은 formula 파생값이다.
- 상태 전환 button은 Notion UI 요소이며, 자동화가 버튼 클릭을 전제로 해서는 안 된다.

### 11-5. 운영 상태 전제
- `첨부 업로드 상태`는 접수 결과가 아니라 후행 첨부 처리 결과를 의미한다.
- `손가락 사진 있음`은 첨부 상태를 기준으로 재판단되는 checkbox write-back 값이다.
- `첨부 최종 확인 완료`는 사람이 체크하되, 첨부 변경 이벤트 발생 시 자동 false 해제가 필요하다.
- `출력 확인 완료`는 발송 전 최종 준비의 필수 확인 요소다.

### 11-6. 운영 화면/뷰 전제
- 실제 운영은 Notion 기반으로 진행한다.
- 운영자는 처리 대기, 발송 대기, 발송 실패, 첨부 분류 대기, 휴지통 첨부 등 운영 흐름을 따라갈 수 있어야 한다.
- 이 뷰 구성의 세부 구현은 TRD/운영 설정 범위에서 구체화한다.
