## 1. 시스템 개요

SawStop Finger Save 시스템은 국내 사용자의 사고 정보를 **한국어 웹폼**으로 접수받고, 접수 즉시 **Notion 사고 페이지**를 생성한 뒤, 첨부는 **R2 + 첨부 DB**로 비동기 처리하고, 운영자는 같은 사고 페이지 본문에서 **영문 리포트**를 작성·수정하며, 최종적으로 **본사 전달까지** 이어지는 운영 구조다.

핵심 기술 목표는 아래와 같다.

1. 접수 성공 기준을 명확히 고정한다.
   - **사고 페이지 본문 저장 성공 = 접수 성공**
2. 첨부 처리와 접수 성공을 분리한다.
   - 첨부 일부 실패, 관리자 알림 실패, 고객 접수증 메일 실패는 접수 실패가 아니다.
3. 데이터 원천과 파생값을 분리한다.
   - 고객 원문은 사고 DB
   - 첨부 메타데이터는 첨부 DB
   - 첨부 원본은 R2
   - 발송 준비 여부는 formula 파생값
4. Notion live schema와 drift 없이 동작하는 보수적 구현을 우선한다.

### 1-1. 1차 완료선
이 프로젝트의 1차 완료선은 **SawStop 본사 전달까지**다. 카트리지 분석, 본사 최종 검토, 결과 회신은 후속 운영 범위다.

### 1-2. 절대 흔들면 안 되는 기준
- 고객 입력 채널은 Notion 폼이 아니라 **별도 웹폼**이다.
- 첨부 원본 기준 저장소는 **Cloudflare R2**다.
- 운영 작업판은 **Notion 사고 페이지**다.
- 영문 리포트는 **같은 사고 페이지 본문**이다.
- 접수 성공 기준은 **본문 저장 성공**이다.
- 손가락 사진은 접수 단계에서 선택이지만 **최종 리포트 전 필수 확보**다.
- 1차는 **단일 Workers 프로젝트** 기준이다.
- 1차 발송은 **수동 기본**이다.

## 3. 주요 데이터 흐름

### 3-1. 접수 기본 흐름
1. 고객이 웹폼에 입력한다.
2. Workers가 `multipart/form-data`를 파싱한다.
3. Workers가 Turnstile을 검증한다.
4. Workers가 입력값을 정규화한다.
5. Workers가 서버 최종 검증을 수행한다.
6. Workers가 `접수번호`를 생성한다.
7. Workers가 사고 DB 페이지의 속성과 접수 시점 기본 본문 블록을 저장한다.
8. 저장 시점에 첨부가 0개면 사고 DB `첨부 업로드 상태 = 완료`, 첨부가 1개 이상이면 `처리중`으로 시작한다.
9. 저장 성공 시 고객에게 `접수 성공 + 접수번호`를 응답한다.
10. 이후 `waitUntil`로 관리자 알림/고객 접수증/첨부 tmp 준비/Queue 적재를 비동기로 시도한다.
11. Consumer가 Queue 메시지를 받아 첨부를 순차 처리한다.

### 3-2. 첨부 후행 흐름
1. 접수 성공 후, 첨부가 있으면 Workers가 Consumer가 참조할 수 있는 tmp 경로를 준비한다.
2. tmp 준비에 성공한 첨부 참조 정보만 Queue 메시지 1건에 담아 적재한다.
3. Consumer가 첨부를 1장씩 순차 처리한다.
4. 각 첨부에 대해 `검증 → tmp 존재 확인 → attachments 경로 반영 → 첨부 DB 기록 → 첨부 DB \`사고건\` relation 연결 → 사고 DB 후행 상태 갱신`을 수행한다.
5. 첨부 DB `R2 Key`에는 **최종 attachments 경로만** 저장한다. tmp 경로는 live 필드에 저장하지 않는다.
6. 모든 첨부 결과를 합산해 사고 DB의 `첨부 업로드 상태`를 갱신한다.
7. 첨부 변경 이벤트에 따라 `손가락 사진 있음`과 `첨부 최종 확인 완료`를 다시 반영한다.

### 3-3. 운영자 검토 흐름
1. 운영자는 사고 페이지를 연다.
2. 입력값, 첨부 유무, 손가락 사진 확보 여부, 카트리지 시리얼 확인 가능 여부를 확인한다.
3. 필요 시 관리자 보완 업로드 페이지로 진입한다.
4. 같은 사고 페이지 본문에서 영문 리포트 초안을 작성한다.
5. 초안 작성 후에도 첨부가 부족하면 관리자 보완 업로드로 **재진입**한다.
6. 보완 첨부 반영 후 다시 같은 사고 페이지 본문으로 복귀해 영문 리포트를 수정한다.
7. 출력 확인, 첨부 확인, 영문 검수 완료를 맞춘 뒤 본사에 수동 전달한다.

### 3-4. 첨부 수명주기 흐름
1. 현재 첨부는 사고건에 relation된 첨부 DB 행과 R2 원본으로 관리한다.
2. 삭제는 휴지통 이동을 기본으로 한다.
3. 7일 복구 가능 기간이 지나면 오전 8시 정리 스케줄 대상이 된다.
4. 만료 휴지통 정리는 FIFO보다 먼저 적용된다.
5. 휴지통 만료 삭제 후에도 저장소가 5GB를 초과하면 FIFO 삭제가 적용될 수 있다.
6. FIFO 삭제는 휴지통을 거치지 않고 첨부 DB row를 `영구삭제` 상태로 정리한다.
7. 삭제/복구/FIFO는 모두 관련 checkbox와 rollup 반영 누락 없이 상태를 다시 계산해야 한다.

## 4. 데이터 모델과 저장 경계

### 4-1. 단일 진실 원천

| 데이터 | 단일 진실 원천 | 비고 |
|---|---|---|
| 고객 구조화 원문 입력값 | 사고 DB 페이지 **속성** | 접수 시점 고객 원문의 원천. 본문 재편집 결과로 덮어쓰지 않음 |
| 사고 페이지 본문 | 영문 리포트 및 운영 편집 영역 | 접수 시점 기본 본문 블록을 만들 수 있으나, 고객 구조화 원문의 단일 진실 원천으로 취급하지 않음 |
| 첨부 원본 파일 | R2 attachments 경로 | Notion files를 원본 저장소로 승격하지 않음 |
| 첨부 메타데이터 | 첨부 DB | 첨부 1개 = 1행 |
| 관리자 알림/접수증 on/off | 운영 설정 DB | 기본 설정 1행만 사용 |
| 접수 식별자 | 사고 DB `접수번호` | 모든 추적의 공통 키 |
| 첨부 개수 파생값 | 사고 DB rollup | 원천 데이터가 아님 |
| 발송 준비 여부 | 사고 DB formula | 직접 쓰기 대상이 아님 |

### 4-2. 저장 경계
- 웹폼 제출 원문은 서버 정규화 전까지 임시 입력값이다.
- 고객 구조화 원문 저장 경계는 사고 DB 속성이다.
- 사고 페이지 본문은 영문 리포트와 운영 편집 경계다.
- 첨부 원본 저장 경계는 R2 attachments 경로다.
- tmp 경로는 후행 첨부 처리를 위한 임시 경계이며, 최종 원본 경계가 아니다.
- 첨부 상태/분류/삭제/표시 순서는 첨부 DB가 경계다.
- formula/rollup/button은 **원천 데이터 쓰기 경로가 아니다**.

### 4-3. 구현 방어선
- relation 쓰기 경로는 **첨부 DB `사고건` relation 1곳**으로 고정한다.
- 사고 DB `첨부 목록`은 읽기/집계 기준으로만 본다.
- rollup 숫자를 코드의 주 원천으로 사용하지 않는다.
- `손가락 사진 있음`과 `첨부 최종 확인 완료`는 formula가 아니라 checkbox write-back이므로, 이벤트 발생 로직이 직접 반영해야 한다.
- 사고 페이지 본문을 수정하더라도 고객 구조화 원문 속성의 source of truth를 본문으로 역전시키지 않는다.
- 첨부 DB `R2 Key`에는 attachments 경로만 저장하고, tmp 경로를 live 필드에 재사용하지 않는다.

- 파일 메타데이터: `파일명`, `R2 Key`, `미리보기 링크`, `썸네일`
- 분류/상태: `첨부 유형`, `상태`, `삭제 사유`, `검수 메모`, `표시 순서`
- 수명주기: `업로드 시각`, `휴지통 이동 시각`, `영구삭제 예정 시각`
- 파생값: `복구 가능`, `현재 카운트용`, `휴지통 카운트용`, `전체 카운트용`

### 5-4. 운영 설정 DB 핵심 규칙
- 기본 설정 1행만 사용한다.
- 비밀번호 등 비밀값은 Workers Secrets로 분리한다.
- 무료 방식의 관리자 알림/고객 접수증 메일 제어에만 사용한다.
- 실패해도 접수 실패로 승격하지 않는다.

## 6. 웹폼 → Workers → Notion → Queue → Consumer → R2 흐름

### 6-1. 요청/응답 단위
- 공개 접수는 `multipart/form-data` 1회 제출이다.
- 고객 응답에는 `page_id` 같은 내부 식별값을 노출하지 않는다.
- 고객 완료 화면에는 내부 첨부 처리 상태를 기본 노출하지 않는다.

### 6-2. 공식 처리 순서
1. `GET /`
   - 고객 웹폼 HTML 서빙
2. `POST /submit`
   - content-type 확인
   - formData 파싱
   - Turnstile 검증
   - normalize
   - validate
   - 접수번호 생성
   - 사고 DB 속성 저장
   - 사고 페이지 기본 본문 블록 저장
   - 첨부 개수에 따라 `첨부 업로드 상태` 초기값 반영
   - 성공 응답 반환
   - `waitUntil` 비동기 시작
3. 비동기 후행
   - 운영 설정 조회 후 관리자 알림 / 고객 접수증 메일 시도
   - 첨부가 있으면 Workers가 tmp 경로를 준비
   - tmp 준비에 성공한 참조 정보 기준으로 Queue 메시지 적재
4. Consumer
   - 첨부별 검증
   - tmp 존재 확인
   - attachments 경로 반영
   - 첨부 DB 기록
   - 첨부 DB `사고건` relation 연결
   - 사고 DB 후행 상태 갱신

### 6-3. 저장 키 규칙
- tmp 경로: `tmp/{접수번호}/{순번}_{업로드시각}.{확장자}`
- 본경로: `attachments/{접수번호}/{순번}_{업로드시각}.{확장자}`
- 순번은 재사용하지 않는다.
- 첨부 DB `R2 Key`에는 **본경로인 attachments key만 저장**한다.
- tmp key는 live DB 필드 전제로 두지 않는다.

### 6-4. Queue payload 기본 원칙
- 메시지는 접수건당 1개다.
- payload는 경량 유지가 원칙이다.
- payload 스키마는 아래로 고정한다.

```json
{
  "version": 1,
  "receiptNumber": "202604031224-2839",
  "pageId": "notion_page_id",
  "attachmentCount": 2,
  "retryCount": 0,
  "attachments": [
    {
      "seq": 1,
      "tmpKey": "tmp/202604031224-2839/0001_20260403122430.jpg",
      "originalFileName": "finger.jpg",
      "contentType": "image/jpeg",
      "sizeBytes": 1234567
    }
  ]
}
```

- `attachments[]`는 `seq / tmpKey / originalFileName / contentType / sizeBytes`만 필수로 가진다.
- 첨부별 참조 정보는 Workers가 tmp를 준비한 뒤 Consumer가 같은 객체를 찾을 수 있는 수준이어야 한다.
- 첨부 원본 바이트나 본문 전체를 payload에 싣지 않는다.

## 7. 웹폼 입력 정규화 / 서버 검증 / Notion 타입 변환 규칙

### 7-1. 검증 3계층

#### A. 프론트 즉시 검증
- 필수값 누락
- 이메일 1차 형식
- 연락처 1차 형식
- 기계 시리얼 1차 형식
- 첨부 형식/용량/개수

#### B. 서버 최종 검증
- Turnstile 검증
- 필수값 재검사
- 정규화 후 형식 재검사
- MIME type / 파일 크기 재검사
- select / multi_select 허용값 재검사

#### C. Notion 타입 변환 검증
- text 입력을 Notion text로 저장할 값인지
- select 옵션명이 live 옵션과 정확히 일치하는지
- multi_select 배열이 허용값과 정확히 일치하는지
- date로 결합되는 값이 유효한지

### 7-2. 정규화 규칙
- 연락처: 숫자만 추출 → 하이픈 정리 → `010-1234-5678`
- 이메일: trim + lowercase
- 회사/학교명: 빈값이면 `NA`
- Saw 시리얼: 공백/기호 제거 + 대문자 변환
- 날짜/시간: `Date of Occurence` 1개 date로 결합. `정확한 시간을 잘 모르겠습니다` 선택 시 선택 날짜의 `12:00 (Asia/Seoul)`로 저장
- 자유서술: trim 중심, 의미 변형 금지
- 첨부 파일명: 원본 파일명 유지 허용

### 7-3. 타입 변환 원칙
- 웹폼 입력 유형과 Notion 저장 유형은 동일할 필요가 없다.
- 자유도 높은 값은 text 저장 우선이다.
- 고정 목록은 select / multi_select 그대로 유지한다.
- 저장 직전 live schema 기준으로 최종 변환한다.

## 8. 접수번호 규칙

### 8-1. 형식
`YYYYMMDDHHmm-연락처뒷4자리`

예: `202604031224-2839`

### 8-2. 역할
- 사고건의 title
- 고객 완료 화면 표시값
- Queue/Consumer/R2 경로/로그의 공통 키

- 중복 회피가 필요하더라도 임의 suffix를 추가하지 않고 open issue 검토가 우선이다.

## 9. 접수 성공 기준과 실패 분리 원칙

### 9-1. 접수 성공 기준
아래가 모두 만족되면 접수 성공으로 본다.
1. 사고 DB 페이지 생성 성공
2. 사고 DB 속성 저장 성공
3. 사고 페이지 본문 저장 성공
4. 사고 DB `상태 = 접수`로 시작

### 9-2. 접수 실패로 보지 않는 항목
- 첨부 일부 실패
- 첨부 전부 실패
- 관리자 알림 실패
- 고객 접수증 메일 실패
- 후행 비동기 재시도 실패

### 9-3. 고객 응답 원칙
- 성공 시:
  - `접수가 완료되었습니다.`
  - `입력하신 내용은 내부 검토 후 확인됩니다.`
  - `접수번호: {접수번호}`
- 실패 시:
  - `접수가 완료되지 않았습니다. 잠시 후 다시 시도해 주세요.`
- 고객 화면에는 내부 오류코드, page_id, 첨부 처리 상태를 노출하지 않는다.

### 9-4. 실패 분리 이유
이 구조는 첨부/R2/메일과 같은 후행 실패가 사고 원문 자체를 죽이지 않게 하려는 것이다. 운영자는 접수된 사고건을 기준으로 이후 보완 업로드와 수동 검토로 복구할 수 있어야 한다.

## 10. 첨부 비동기 처리 규칙

### 10-1. 첨부 기본 정책
- 최대 4장
- 파일당 10MB 이하
- 허용 형식: `jpg`, `jpeg`, `png`, `webp`, `heic`, `heif`
- 원본 파일명 유지 허용
- 내부 저장 키는 별도 생성
- 형식/용량 오류는 해당 파일만 제외

### 10-2. 처리 순서
첨부 1장별 처리 순서는 아래와 같다.
1. Workers가 Queue가 참조할 수 있는 tmp 경로를 준비한다.
2. Consumer가 파일 검증을 수행한다.
3. Consumer가 tmp 저장 존재를 확인한다.
4. Consumer가 attachments 경로를 최종 원본 위치로 반영한다.
5. Consumer가 첨부 DB 1행을 기록한다.
6. Consumer가 첨부 DB `사고건` relation을 연결한다.
7. Consumer가 사고 DB 후행 상태를 갱신한다.

### 10-3. `R2 Key` 저장 규칙
- 첨부 DB live 필드 `R2 Key`에는 **최종 attachments 경로만** 저장한다.
- tmp 경로는 후행 처리용 임시 경로이며, 첨부 DB `R2 Key`의 값으로 사용하지 않는다.
- tmp key는 live DB 별도 필드로 저장하지 않는다.

### 10-3A. `첨부 ID` / `표시 순서` 고정 규칙
- `첨부 ID`는 `ATT-{receiptNumber}-{seq4}` 형식으로 생성한다.
- 예: `ATT-202604031224-2839-0001`
- `seq4`는 동일 사고건 내 표시 순서 4자리 zero-pad 값이다.
- `표시 순서`는 고객 웹폼 선택 순서 그대로 1,2,3... 부여한다.
- 관리자 보완 업로드는 기존 최대값 다음 순번부터 이어 붙인다.
- 삭제/휴지통/복구/FIFO 후에도 순번은 재사용하지 않는다.

### 10-3B. `미리보기 링크` / `썸네일` 고정 규칙
- `미리보기 링크`는 R2 접근 URL(서명 URL 또는 Worker 프록시 URL)로 저장한다.
- 1차 MVP에서 `썸네일`은 `미리보기 링크`와 동일 URL 저장을 허용한다.
- 별도 썸네일 이미지 변환 파이프라인은 1차 범위에 넣지 않는다.

### 10-4. 재시도 규칙
- 파일별 최대 2회 자동 재시도
- 그래도 실패하면 관리자 수동 보완 업로드 대상으로 넘긴다.
- 고객이 다시 제출하지 않아도 접수는 유지한다.

### 10-5. `첨부 업로드 상태` 갱신 규칙

| 상황 | 값 |
|---|---|
| 첨부 0개 | `완료` |
| 첨부 1개 이상, 본문 저장 직후 | `처리중` |
| 모든 첨부 성공 | `완료` |
| 일부 첨부 실패 | `일부 실패` |
| 모든 첨부 실패 | `실패` |

### 10-6. 첨부 분류 원칙
- 1차 범위에서 AI 자동분류를 사용하지 않는다.
- `첨부 유형`은 병준이 직접 지정한다.
- 관리자 업로드는 업로드 시점에 유형을 즉시 선택한다.
- 고객 웹폼 첨부는 첨부 DB 생성 시점에 `첨부 유형=null`로 저장한다.
- 고객 웹폼 첨부는 `첨부 분류 대기` 뷰에서 병준이 수동 분류한다.
- `손가락 사진 있음` 계산과 발송 준비 판단에는 `첨부 유형=손가락 사진`으로 분류가 끝난 현재 첨부만 반영한다.

## 11. 관리자 수동 보완 업로드 흐름

### 11-1. 진입 방식
- 공개 고객 웹폼과 분리된 별도 관리자 업로드 페이지를 사용한다.
- 같은 도메인 내 숨은 경로 1개를 사용한다.
- 사고 페이지 상단 `보완 첨부 업로드` 버튼에서 새 탭으로 연다.
- `?receiptNumber={접수번호}` 방식의 자동 포함 링크를 지원한다.

### 11-2. 인증 규칙
- 비밀번호 1개
- Turnstile 검증
- session cookie 기반 인증 유지
- 브라우저 종료까지 유지
- 5회 실패 시 10분 잠금

### 11-3. 검색 규칙
- 검색 대상 상태: `접수 / 진행중 / 반려`
- 완료건 제외
- 접수번호 우선 검색
- 없으면 연락처 검색
- 숫자 4자리 입력 시 연락처 뒷4자리 + 접수번호 뒷4자리 동시 검색
- 최근 접수일시순 최대 20건
- 결과 1건이면 자동 선택

### 11-4. 화면 고정 표시 사고건 요약
- `접수번호`
- `상태`
- `사고 발생일시`
- `연락처`
- `기계 시리얼 번호`

### 11-5. 업로드 규칙
- 웹폼과 같은 첨부 제한을 사용한다.
- 첨부 유형은 업로드 시점에 반드시 선택한다.
- 허용값은 `손가락 사진 / 브레이크 카트리지 사진 / 기타` 3개만 사용한다.

### 11-6. 결과 구분
- `성공`
- `부분 성공`
- `실패`
- `인증 실패`

### 11-7. 중복/누락 방어선
- 관리자 업로드도 첨부 1개 = 첨부 DB 1행 원칙을 유지한다.
- 사고건 relation 연결 누락이 생기면 사고 페이지에서 첨부가 보이지 않으므로, 업로드 완료 판정은 R2 저장만으로 끝내지 않는다.
- 부분 성공 시 실패 파일명을 로그에 반드시 남긴다.

## 12. 영문 리포트 작성 / 출력 / 발송 준비 흐름

### 12-1. 저장 위치
- 별도 영문 리포트 DB는 만들지 않는다.
- 각 사고건 Notion 페이지 본문에서 직접 작성/수정한다.

### 12-2. 외부 제목
`Report a Save (Known or Suspected Finger Contact)`

### 12-3. 템플릿 원칙
- 항목명은 Notion DB의 영문 속성명을 그대로 사용한다.
- `Phone`, `Email`, `Consent for Promotional Use`는 고정 포함 항목이다.
- 값은 항목명 아래 줄에 배치한다.
- 긴 항목명은 자동 줄바꿈을 허용한다.

### 12-4. 영문화 모드
- `완전 영문화`
- `규칙/공식명 영문화(번역 판단 필요 내용 원문 유지)`

구분 원칙:
- 규칙 변환 가능한가
- 공식 영문명 확인 가능한가
- 번역 판단이 필요한가

### 12-5. 규칙 변환 예
- `예 (YES)` → `YES`
- `동의 (YES)` → `YES`
- `보통 (Normal)` → `Normal`
- `푸시스틱 (Push Stick)` → `Push Stick`

### 12-6. 번역 판단이 필요한 값
- 고객 자유서술
- 사고 원인 설명
- 사고 당시 상황 설명
- 예상 피해 설명
- 그 외 문장형 설명

### 12-7. 초안 작성 후 보완 업로드 재진입
- 영문 초안 작성 후에도 첨부가 부족하면 관리자 보완 업로드로 다시 들어갈 수 있어야 한다.
- 보완 업로드 후에는 같은 사고 페이지 본문으로 복귀해 초안을 계속 수정해야 한다.
- 이 재진입 루프는 예외 흐름이 아니라 원문 워크플로우에 포함된 정상 운영 흐름으로 본다.

### 12-8. 출력 원칙
- 웹뷰와 PDF는 **같은 영문 리포트 원본 본문**을 기준으로 한다.
- 출력은 발송 준비 전 최종 확인 흐름의 일부다.
- 1차 범위에서는 가독성과 동일 원본 유지가 우선이며, 별도 문서 원천을 만들지 않는다.

### 12-9. 발송 준비 완료 판단 규칙
`발송 준비 완료(자동)`은 아래 3개가 모두 true일 때만 true다.
- `영문 검수 완료`
- `첨부 최종 확인 완료`
- `출력 확인 완료`

### 12-10. 발송 운영 원칙
- 1차는 `발송 방식 = 수동` 기본이다.
- SMTP 세팅 완료 후에만 `반자동` 확장을 연다.
- 별도 발송 상태 DB는 만들지 않는다.
- 발송 상태는 `발송 준비 완료(자동)` + `발송 완료 시각` + `발송 실패 메모` 조합으로 판단한다.

## 13. 상태값 및 주요 판정 규칙

### 13-1. 사고 DB 상태 전제
- 신규 접수 시 `상태 = 접수`
- 관리자 검색 대상은 `접수 / 진행중 / 반려`
- 완료건은 관리자 업로드 검색에서 제외
- 사고 DB `상태`의 전체 live 옵션 목록은 미확정이므로 open issue

### 13-2. 첨부 DB 상태 전제
- 현재 문맥상 계산에 필요한 값은 `현재 / 휴지통 / 영구삭제`
- 전체 live status 옵션 목록은 미확정이므로 open issue

### 13-3. `손가락 사진 있음` 규칙
현재 첨부 목록 안에서 아래 조건을 만족하는 첨부가 1개 이상이면 true다.
- `첨부 유형 = 손가락 사진`
- `상태 = 현재`

아래 상태는 계산에서 제외한다.
- `휴지통`
- `영구삭제`
- 그 외 삭제 처리 상태

### 13-4. `첨부 최종 확인 완료` 규칙
- 사람의 최종 검토 체크다.
- `손가락 사진 있음 = true`여야 true 가능하다.
- 아래 이벤트가 발생하면 자동 false 해제해야 한다.
  - 새 첨부 추가
  - 유형 변경
  - 삭제
  - 휴지통 이동
  - 복구
  - 영구삭제/FIFO
  - 마지막 손가락 사진 상실

### 13-5. 휴지통 / FIFO 규칙
- 일반 삭제는 휴지통 이동
- 휴지통 첨부는 7일 복구 가능
- 원래 사고건으로만 복구 가능
- 다음 오전 8시 정리 스케줄에서 만료 삭제
- 만료 휴지통 정리는 FIFO보다 먼저 적용
- 휴지통 만료 삭제 후에도 5GB 초과 시 FIFO 적용
- FIFO 삭제는 휴지통을 거치지 않음
- FIFO 삭제 시 첨부 DB `상태 = 영구삭제`로 정리
- 필요 시 `삭제 사유` 또는 `검수 메모`로 이력을 남긴다.
- `check:fifo-trash-candidates` 같은 live-read 검증은 deterministic parity, scenario execution, baseline, CI, product wiring과 분리된 수동 확인 경계로 유지한다.
- 오전 8시 정리 스케줄의 실제 cleanup owner, 5GB 판단에 포함할 R2/storage population, `영구삭제 예정 시각` 계산식은 아직 구현 기준으로 닫지 않는다.

## 14. relation / rollup / formula / button 사용 전제와 구현상 제약

### 14-1. relation
- 사고 DB `첨부 목록`은 첨부 집합의 읽기/집계 기준이다.
- 첨부 DB `사고건` relation은 반드시 원래 사고건 1개에 연결된다.
- 코드 쓰기 경로는 첨부 DB `사고건` relation 1곳으로 고정한다.
- relation 연결 실패 시 R2 원본이 존재해도 사고 페이지에서 첨부가 누락된다.

### 14-2. rollup
- `현재 첨부 개수`
- `휴지통 첨부 개수`
- `전체 첨부 개수`

제약:
- 직접 저장/수정 대상이 아니다.
- relation과 첨부 DB 상태가 맞아야 정확하다.
- 화면 표시에는 써도, 최종 판단은 원천 relation/첨부 상태 확인이 가능해야 한다.
- API 조회 시 축약/pagination 이슈를 고려한다.

### 14-3. formula
- 사고 DB 대표 formula: `발송 준비 완료(자동)`
- 첨부 DB 대표 formula: `복구 가능`, `현재 카운트용`, `휴지통 카운트용`, `전체 카운트용`

제약:
- 직접 업데이트 대상이 아니다.
- 입력값이 아니라 파생 판단 결과다.
- 원천 속성이 흔들리면 formula도 같이 틀어진다.

### 14-4. button
- `접수->진행중`
- `접수->반려`
- `진행중->완료`

제약:
- 운영용 UI 전환 도구로 본다.
- 자동화가 button 클릭 자체를 API 전제 조건으로 삼아서는 안 된다.
- 진실 원천은 button 이름이 아니라 실제 `상태` 값과 관련 날짜/검수 값이다.
- button이 상태 외에 무엇을 함께 수정하는지는 미확정이다.

## 15. 데이터 정합성 규칙

### 15-1. 정합성 기본 원칙
1. live schema 우선
2. 후보 속성 전제 금지
3. 원천값과 파생값 분리
4. 접수 성공과 후행 실패 분리
5. relation 단일 쓰기 경로 고정
6. 첨부 변경 이벤트 후 checkbox/파생값 재반영
7. 고객 구조화 원문 속성과 영문 리포트 본문 source of truth 분리
8. 첨부 DB `R2 Key`에는 attachments 경로만 저장

### 15-2. 주요 위험과 방어선

| 위험 | 발생 지점 | 예방 | 탐지 |
|---|---|---|---|
| schema drift | Notion 속성명/타입 변경 | live schema 기준 대조 후 저장, 후보 속성 미사용 | 저장 실패 로그, 옵션 불일치 오류, 배포 전 schema 점검 |
| 잘못된 select 값 | 서버 → Notion 저장 시 | live 허용값 정확 일치 검증 | `Value is not a valid select option`류 오류 추적 |
| relation 누락 | Consumer / 관리자 업로드 | 첨부 DB `사고건` relation 쓰기 필수화 | R2 저장 성공인데 사고 페이지 첨부 누락 시 검출 |
| 후행 비동기 실패 | tmp 준비 / Queue / Consumer | 접수 성공과 분리, 파일별 재시도 2회 | `첨부 업로드 상태`, 로그, 오류 플래그/사유 후보 |
| 수동 보완 업로드 중복/누락 | 관리자 업로드 | 첨부 1개=1행 원칙, 결과를 성공/부분성공 분리 | `failedFileNames`, 사고 페이지 첨부 수/목록 점검 |
| 휴지통/FIFO 반영 누락 | 삭제/복구/정리 스케줄 | 상태 전환과 checkbox false reset을 이벤트 기반으로 반영 | `손가락 사진 있음`/`첨부 최종 확인 완료`와 실제 첨부 불일치 점검 |
| rollup 오판 | 화면/운영 판단 | rollup을 참고값으로만 사용 | 원천 relation/첨부 DB 상태 재조회 |
| 고객 원문과 본문 혼선 | 영문 리포트 편집 / 출력 / 재저장 | 고객 구조화 원문은 사고 DB 속성 기준으로 고정 | 속성과 본문 값 충돌 시 source of truth 점검 |
| 미분류 첨부 누락 | 고객 웹폼 첨부 후행 처리 | 분류 전 저장 방식 확정 전까지 발송 준비 판단에 직접 사용하지 않음 | `첨부 분류 대기` 뷰 운영 |
| tmp/본경로 혼선 | R2 저장/재시도 | Workers는 tmp 준비, Consumer는 attachments 반영, `R2 Key`는 attachments만 저장 | `R2 Key`가 tmp 경로로 저장됐는지 로그/샘플 점검 |

### 15-3. 정합성 판정 기준
- 사고 원문 속성이 Notion에 저장되지 않으면 접수 실패다.
- 사고 페이지 본문 저장이 실패하면 접수 실패다.
- 첨부 원본이 tmp에만 있고 attachments/R2 Key/relation이 확정되지 않았다면 **첨부 저장 성공으로 보지 않는다**.
- 첨부 원본이 R2에만 있고 relation이 없으면 **첨부 저장 성공으로 보지 않는다**.
- `손가락 사진 있음`이 false인데 실제 현재 첨부에 손가락 사진이 존재하면 정합성 오류다.
- `첨부 최종 확인 완료`가 true인데 마지막 손가락 사진이 사라졌다면 정합성 오류다.
- `발송 준비 완료(자동)`을 사람이 직접 수정하려는 흐름은 허용하지 않는다.

## 16. 예외 처리 정책

### 16-1. 고객 화면
- 실패 시 일반 실패 문구만 보여준다.
- 관리자 추적용 내부 오류코드는 노출하지 않는다.

### 16-2. 운영 처리
- 1차 원칙은 `서버/호스팅 로그 + 관리자 이메일 알림`이다.
- Notion 오류 DB, 문자, 카카오톡 등은 1차 범위에 넣지 않는다.

### 16-3. 오류 위치 분류
- `입력값 확인`
- `파일 업로드`
- `Notion 저장`
- `이메일 알림`

### 16-4. 예외 처리 원칙
- 가능한 한 시스템 원문 메시지를 남긴다.
- 사용자가 복구 가능한 경우 접수 실패로 올리지 않는다.
- 접수 실패와 운영 후속 조치 필요를 명확히 구분한다.

### 16-5. 대표 위험 항목
- `Date of Occurence` 시간 미상 저장 방식은 `12:00 (Asia/Seoul)`로 잠겼다.
- 4자리 검색 오탐/다건 충돌 가능성
- formula / rollup 조회 오차 가능성
- 후보 속성 오남용 위험
- 고객 웹폼 첨부는 `첨부 유형=null`로 저장하고 `첨부 분류 대기` 뷰에서 수동 분류한다.

## 18. 구현 페이즈

### Phase 1. 접수 성공 닫기
목표:
- 공개 웹폼 제출 → 정규화/검증 → 접수번호 생성 → 사고 DB 속성/기본 본문 저장 → 완료 화면까지 닫는다.

완료 기준:
- 사고 DB 페이지 생성 성공
- 사고 DB 속성 저장 성공
- 사고 페이지 기본 본문 저장 성공
- 고객 완료 화면에 접수번호 표시
- 접수 실패 시 일반 실패 문구 표시

### Phase 2. 첨부 비동기 닫기
목표:
- Workers tmp 준비, Queue 적재, Consumer 처리, 첨부 DB 기록, relation 연결, `첨부 업로드 상태` 갱신을 닫는다.

완료 기준:
