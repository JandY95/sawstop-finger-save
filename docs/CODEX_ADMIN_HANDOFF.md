# SAWSTOP Finger Save | CODEX_ADMIN_HANDOFF

## 0. 문서 목적
이 문서는 SAWSTOP Finger Save 저장소에서 Codex가 작업을 시작하기 전에,
현재 구현 상태, 현재 운영 기준, 다음 작업 원칙을 빠르게 파악하도록 돕는 운영자용 handoff 문서다.

이 문서는 새 기획 문서가 아니다.
이 문서는 source of truth를 대체하지 않는다.
이 문서는 현재 저장소에서 Codex가 흔들리지 않도록 돕는 “운영 기준 요약 + 현재 상태판”이다.

---

## 1. 이 문서를 읽는 Codex의 기본 태도

너는 이 프로젝트를 설명만 받는 조수가 아니라,
실제로 SAWSTOP Finger Save를 직접 운영하는 관리자라고 생각하고 판단한다.

즉 아래 기준으로 움직인다.

- 고객이 실제로 접수에 성공해야 한다.
- 첨부가 일부 실패해도 사고 접수 자체는 살아 있어야 한다.
- Notion live schema와 어긋나는 편의 구현을 하지 않는다.
- 자동화보다 운영 복원성과 정합성을 우선한다.
- 한 번에 하나의 흐름만 바꾼다.
- 추측으로 새 필드, 새 상태, 새 자동화를 넣지 않는다.

## 3. 절대 흔들면 안 되는 운영 기준

### 3-1. 접수 / 저장
- 고객 입력 채널은 Notion 폼이 아니라 별도 웹폼이다.
- 접수 성공 기준은 `사고 DB 페이지 생성 + 매핑 대상 속성 저장 + 기본 본문 저장 성공`이다.
- 첨부 실패는 접수 실패와 분리한다.
- 고객 완료 화면에는 접수번호만 명확히 보여주고 내부 상태는 기본 노출하지 않는다.

### 3-2. 첨부 / 저장소
- 첨부 원본 저장소는 Notion file이 아니라 Cloudflare R2다.
- 사고 DB `첨부(선택)` file 속성은 MVP 기준 자동 원본 저장소가 아니다.
- 첨부 메타데이터는 `SAWSTOP 첨부 관리` DB가 원천이다.
- 첨부와 사고의 연결 write 경로는 첨부 DB `사고건` relation 1곳으로 유지한다.
- `R2 Key`에는 최종 `attachments/...` 키만 저장한다.
- `tmp/...`는 임시 경로일 뿐 최종 키처럼 취급하지 않는다.
- `첨부 ID`는 `ATT-{receiptNumber}-{seq4}` 형식을 유지한다.
- `표시 순서`는 고객 업로드 순서 1,2,3...로 시작하고 관리자 업로드는 현재 최대값 다음부터 이어간다.
- 삭제/휴지통/복구/FIFO 후에도 표시 순서를 재사용하지 않는다.
- `미리보기 링크`는 MVP에서 R2 접근 URL을 사용한다.
- `썸네일`은 MVP에서 동일 URL 재사용을 허용한다.

### 3-3. 영문 리포트 / 발송
- 영문 리포트는 별도 DB가 아니라 같은 사고 페이지 본문에서 관리한다.
- 1차 발송은 수동이 기본이다.
- `발송 준비 완료(자동)`은 파생값으로 다루며 직접 쓰기 대상이 아니다.

### 3-4. 손가락 사진 / 첨부 검토
- 손가락 사진은 접수 시점 필수는 아니다.
- 하지만 최종 발송 전에는 반드시 확보해야 한다.
- 고객 웹폼은 첨부 유형을 묻지 않는다.
- 고객 업로드 첨부는 생성 시 `첨부 유형 = null`이다.
- 관리자 보완 업로드는 업로드 시점에 첨부 유형을 즉시 선택한다.
- 1차 범위에서 AI 첨부 자동분류를 넣지 않는다.

## 4. 현재 구현 상태 요약

현재 저장소 기준으로 아래 흐름은 이미 구현되어 있거나 최소 뼈대가 올라와 있다.

### 4-1. 고객 웹폼
- `GET /`에서 고객 웹폼을 렌더링한다.
- 고객 폼은 별도 렌더 함수에서 구성된다.
- 폼 제출은 `POST /submit`으로 보낸다.
- 성공 시 고객 완료 화면에 접수번호를 보여주는 구조다.

### 4-2. 접수 처리
- `handleSubmit()`에서 `multipart/form-data`를 받는다.
- 입력값 정규화/검증 후 접수번호를 만든다.
- 사고 DB 속성을 만들고 사고 페이지를 생성한다.
- 기본 본문을 저장한다.
- 첨부가 있으면 비동기로 첨부 처리 흐름을 시작한다.

### 4-3. 첨부 비동기 처리
- 접수 시 첨부는 먼저 tmp R2에 올린다.
- 업로드된 첨부 참조 정보를 queue payload로 적재한다.
- consumer는 queue 메시지를 받아 첨부를 순차 처리한다.
- consumer는 tmp를 final R2로 반영하고 첨부 DB 페이지를 보장한다.
- 처리 결과에 따라 사고건 `첨부 업로드 상태`를 갱신한다.

### 4-4. 손가락 사진 있음 / 첨부 재평가
- `손가락 사진 있음`은 write-back checkbox로 운영한다.
- 조건은 `같은 사고건 relation + 첨부 유형 = 손가락 사진 + 상태 = 현재` 인 첨부가 1개 이상일 때다.
- 휴지통/영구삭제 첨부는 계산에서 제외한다.
- 관리자 첨부 유형 변경 시 재계산한다.
- 휴지통 이동 시 재계산한다.
- 복구 시 재계산한다.
- FIFO 또는 마지막 손가락 사진 상실도 재평가 대상으로 본다. 이 경로의 실제 구현 상태는 별도 점검 대상으로 유지한다.
- 첨부 관련 변화가 있을 때 `첨부 최종 확인 완료`는 다시 false로 해제될 수 있어야 한다.

### 4-5. 현재 구현 상태 해석
- 현재 구조는 “자동 이미지 인식”이 아니다.
- 현재 구조는 `수동 분류 기반 자동 재계산`이다.
- 즉, 고객 첨부가 처음 들어왔다고 바로 `손가락 사진 있음=true`가 되는 구조가 아니라,
  관리자가 첨부 유형을 올바르게 분류하면 그 뒤 판정은 자동으로 따라가는 구조다.

## 5. 현재 운영 판단

### 5-1. 지금 채택한 운영 방식
현재 운영 방식은 아래로 잠근다.

- 손가락 사진 판정은 자동 이미지 인식이 아니다.
- 현재 방식은 `관리자 수동 분류 + 시스템 자동 재계산`이다.
- 이 방식은 현재 코드/문서/운영 기준과 가장 잘 맞는다.
- 당장은 자동 분류를 추가하지 않는다.
- 당장은 분류 보조 장치도 추가하지 않는다.
- 지금은 더 똑똑한 시스템보다 덜 헷갈리는 시스템을 우선한다.

### 5-2. 왜 이렇게 가는가
- 사고/증빙 데이터는 오판정 비용이 크다.
- 현재 구조는 이미 수동 분류 이후 재계산이 자동으로 닫혀 있다.
- 성급한 자동 분류는 편해 보여도 운영 책임을 높인다.
- 따라서 지금은 정확성 우선으로 간다.

## 10. 현재 손가락 사진 판정 기준

현재 저장소 기준 운영 해석은 아래와 같이 잠근다.

- 고객 첨부는 처음부터 손가락 사진으로 확정되지 않는다.
- 첨부 페이지 생성 시 고객 첨부는 `첨부 유형=null` 상태로 시작할 수 있다.
- 관리자가 첨부 유형을 `손가락 사진`으로 지정하면 `손가락 사진 있음`을 재계산한다.
- 휴지통 이동/복구/FIFO 등 첨부 상태 변화가 생기면 다시 재계산한다.
- 따라서 현재 손가락 사진 판정 구조는
  `자동 이미지 판독`이 아니라
  `수동 분류 기반 자동 재계산`이다.

이 기준은 현재 운영자 판단상 유지한다.

## 13. 운영자 최종 원칙

이 저장소에서 가장 중요한 것은
“한 번 더 자동화하는 것”이 아니라
“실제 사고 접수와 후속 정리가 덜 꼬이게 만드는 것”이다.

Codex는 항상 아래 질문을 먼저 스스로 확인한다.

- 이 수정이 고객 접수 성공을 흔드는가?
- 이 수정이 첨부 원본 / R2 / Notion 정합성을 흔드는가?
- 이 수정이 손가락 사진 판정의 신뢰도를 떨어뜨리는가?
- 이 수정이 운영자가 사고건을 복원하기 어렵게 만드는가?

하나라도 위험하면, 먼저 구현하지 말고 기준 충돌 여부를 보고한다.
