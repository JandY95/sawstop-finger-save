# MVP_CHECKLIST.md

이 문서는 SawStop Finger Save MVP 완료 판정 체크리스트다.
문서에 없는 기능은 포함하지 않는다.
각 항목은 실제로 확인 가능한 조건만 사용한다.

## 1. 고객 접수

### 1.1 공개 웹폼 접근
- 체크 항목: 공개 고객 웹폼이 단일 페이지 구역형으로 열리고 7개 구역 순서가 문서와 일치한다.
- 완료 기준: `연락받으실 정보 -> 사고가 발생한 때와 사람 -> 손가락과 상처 정보 -> 기계 및 카트리지 정보 -> 작업 당시 정보 -> 사고 설명 -> 사진 첨부 및 동의` 순서가 그대로 보인다.
- 검증 방법: 브라우저에서 `/` 접속 후 화면 순서, 상단 제목, 안내문, 제출 버튼 문구를 확인한다.
- 실패 시 먼저 볼 파일: `src/render.ts`, `src/index.ts`

### 1.2 필수 입력 프론트 검증
- 체크 항목: 필수 항목 누락 시 제출이 차단되고 첫 오류 필드로 이동한다.
- 완료 기준: 연락처, 이메일, 사고 발생일, 발생 시간 또는 시간 미상 체크, 신체 접촉 부위, 기계 시리얼 번호, 사고 설명, 홍보 활용 동의 누락 시 제출 성공 응답이 나오지 않는다.
- 검증 방법: 필수 항목을 하나씩 비우고 제출해 오류 문구와 포커스 이동을 확인한다.
- 실패 시 먼저 볼 파일: `src/render.ts`, `src/validate.ts`

### 1.3 완료 화면 비노출 규칙
- 체크 항목: 접수 성공 후 완료 화면에 내부 식별값과 내부 상태가 노출되지 않는다.
- 완료 기준: 성공 화면에는 `접수가 완료되었습니다.`, `입력하신 내용은 내부 검토 후 확인됩니다.`, `접수번호`만 보이고 `page_id`, `첨부 업로드 상태`, 내부 오류코드, 관리자 링크가 보이지 않는다.
- 검증 방법: 정상 제출 후 성공 화면 텍스트와 DOM을 확인한다.
- 실패 시 먼저 볼 파일: `src/render.ts`, `src/index.ts`

### 1.4 고객 웹폼 첨부 유형 비노출
- 체크 항목: 고객 웹폼에는 첨부 유형 선택 UI가 없다.
- 완료 기준: 고객 화면에 `손가락 사진`, `브레이크 카트리지 사진`, `기타` 중 하나를 고르는 입력 요소가 없다.
- 검증 방법: 웹폼 첨부 구역 UI와 DOM을 확인한다.
- 실패 시 먼저 볼 파일: `src/render.ts`

### 1.5 시간 미상 UI
- 체크 항목: 고객 웹폼에 `정확한 시간을 잘 모르겠습니다` 체크 UI가 있다.
- 완료 기준: 체크박스를 선택하면 발생 시간 입력이 비활성화되고 제출 자체는 허용된다.
- 검증 방법: 웹폼에서 체크박스 선택 전후의 시간 입력 상태를 확인한다.
- 실패 시 먼저 볼 파일: `src/render.ts`, `src/validate.ts`

## 2. 사고 DB 저장

### 2.1 접수번호 생성
- 체크 항목: 접수 성공 시 접수번호가 지정 형식으로 생성된다.
- 완료 기준: 접수번호가 `YYYYMMDDHHmm-연락처뒷4자리` 형식을 만족한다.
- 검증 방법: `scripts/smoke-submit.ts` 실행 또는 수동 제출 후 반환된 접수번호를 확인한다.
- 실패 시 먼저 볼 파일: `src/receipt.ts`, `src/index.ts`

### 2.2 사고 DB 속성 저장
- 체크 항목: 접수 시 사고 DB 페이지가 1개 생성되고 매핑 대상 속성이 저장된다.
- 완료 기준: 사고 DB에서 새 페이지 1개가 생성되고 `접수번호`, `상태`, `Date of Occurence`, `Phone`, `Email`과 필수 매핑 속성이 비어 있지 않다.
- 검증 방법: 제출 후 Notion 페이지를 열어 저장된 속성을 확인한다.
- 실패 시 먼저 볼 파일: `src/mapper.ts`, `src/notion.ts`, `src/index.ts`

### 2.3 기본 본문 저장
- 체크 항목: 접수 성공 판정 전에 사고 페이지 기본 본문이 저장된다.
- 완료 기준: 사고 DB 페이지가 생성되었더라도 기본 본문 저장이 실패하면 접수 성공 응답이 반환되지 않는다.
- 검증 방법: `scripts/smoke-submit.ts` 또는 로그로 사고 페이지 생성과 본문 저장이 모두 성공한 뒤에만 성공 응답이 나오는지 확인한다.
- 실패 시 먼저 볼 파일: `src/notion.ts`, `src/index.ts`

### 2.4 Date of Occurence 시간 미상 저장
- 체크 항목: 시간 미상 체크 사용 시 `Date of Occurence`가 잠금 규칙대로 저장된다.
- 완료 기준: `정확한 시간을 잘 모르겠습니다` 선택 시 `Date of Occurence`가 선택한 날짜의 `12:00 Asia/Seoul` 기준 값으로 저장된다.
- 검증 방법: 시간 미상 체크로 제출 후 Notion 저장값을 확인한다.
- 실패 시 먼저 볼 파일: `src/normalize.ts`, `src/mapper.ts`, `src/notion.ts`

## 3. 첨부 비동기 처리

### 3.1 첨부 0건 처리
- 체크 항목: 첨부 없이 접수해도 접수가 성공하고 업로드 상태가 `완료`로 저장된다.
- 완료 기준: 첨부 0건 제출 후 고객은 성공 화면을 받고, 사고 DB `첨부 업로드 상태`는 `완료`다.
- 검증 방법: `node scripts/smoke-submit.ts` 또는 수동 제출 후 Notion 상태를 확인한다.
- 실패 시 먼저 볼 파일: `src/index.ts`, `src/mapper.ts`, `src/notion.ts`

### 3.2 Queue payload 고정 스키마
- 체크 항목: 첨부가 있는 접수는 잠긴 payload 스키마로 Queue에 적재된다.
- 완료 기준: payload에 `version`, `receiptNumber`, `pageId`, `attachmentCount`, `retryCount`, `attachments[].seq`, `attachments[].tmpKey`, `attachments[].originalFileName`, `attachments[].contentType`, `attachments[].sizeBytes`가 존재한다.
- 검증 방법: 로그 또는 Queue 직전 payload 출력으로 필드 구성을 확인한다.
- 실패 시 먼저 볼 파일: `src/queue.ts`, `src/types.ts`, `src/index.ts`

### 3.3 첨부 DB 1행 생성과 relation 연결
- 체크 항목: 첨부 1개당 첨부 DB 1행이 생성되고 원래 사고건 relation이 연결된다.
- 완료 기준: 첨부 N개 제출 시 첨부 DB에 N행이 생성되고 각 행의 `사고건` relation이 동일 사고 페이지를 가리킨다.
- 검증 방법: 제출 후 첨부 DB 행 수와 relation 대상 페이지를 확인한다.
- 실패 시 먼저 볼 파일: `src/consumer.ts`, `src/notion.ts`

### 3.4 R2 Key 최종 경로 저장
- 체크 항목: 첨부 DB `R2 Key`에 tmp가 아니라 attachments 최종 경로가 저장된다.
- 완료 기준: `R2 Key` 값이 `attachments/{receiptNumber}/...` 형식이고 `tmp/`로 시작하지 않는다.
- 검증 방법: 첨부 DB에서 생성된 행의 `R2 Key`를 직접 확인한다.
- 실패 시 먼저 볼 파일: `src/consumer.ts`, `src/notion.ts`

### 3.5 일부 실패 분리
- 체크 항목: 첨부 일부 실패가 접수 성공을 깨지 않고 상태만 `일부 실패`로 남는다.
- 완료 기준: 첨부 일부가 실패해도 고객은 접수 성공 응답을 받고, 사고 DB `첨부 업로드 상태`는 `일부 실패`다.
- 검증 방법: 실패 유도 입력으로 제출 후 고객 응답과 Notion 상태를 동시에 확인한다.
- 실패 시 먼저 볼 파일: `src/consumer.ts`, `src/notion.ts`, `src/index.ts`

## 4. 관리자 보완 업로드

### 4.1 관리자 인증
- 체크 항목: 관리자 업로드 페이지는 인증 없이 접근되지 않는다.
- 완료 기준: 비밀번호와 Turnstile 검증 없이 업로드 화면에 진입할 수 없고, 5회 실패 시 10분 잠금이 동작한다.
- 검증 방법: 인증 실패 5회와 인증 성공 1회를 각각 재현한다.
- 실패 시 먼저 볼 파일: `src/admin-auth.ts`, `src/index.ts`, `src/turnstile.ts`

### 4.2 완료건 제외 검색
- 체크 항목: 관리자 검색에서 완료건이 제외된다.
- 완료 기준: 상태가 `완료`인 사고건은 접수번호 검색, 연락처 검색, 4자리 검색 결과에 포함되지 않는다.
- 검증 방법: 완료건과 미완료건 샘플로 검색 결과를 비교한다.
- 실패 시 먼저 볼 파일: `src/admin-search.ts`, `src/notion.ts`, `src/schema.ts`

### 4.3 보완 업로드 저장 구조
- 체크 항목: 관리자 업로드도 고객 첨부와 같은 저장 구조를 사용한다.
- 완료 기준: 업로드한 파일이 R2 저장, 첨부 DB 1행 생성, `사고건` relation 연결, `표시 순서` 증가를 모두 만족한다.
- 검증 방법: `node scripts/smoke-admin-upload.ts` 실행 또는 수동 업로드 후 첨부 DB를 확인한다.
- 실패 시 먼저 볼 파일: `src/admin-upload.ts`, `src/consumer.ts`, `src/notion.ts`

### 4.4 관리자 업로드 즉시 유형 지정
- 체크 항목: 관리자 업로드는 첨부 유형을 업로드 시점에 지정한다.
- 완료 기준: 관리자 업로드로 생성된 첨부 DB 행의 `첨부 유형`은 `손가락 사진`, `브레이크 카트리지 사진`, `기타` 중 하나이며 null이 아니다.
- 검증 방법: 유형별로 1건씩 업로드하고 첨부 DB 값을 확인한다.
- 실패 시 먼저 볼 파일: `src/admin-upload.ts`, `src/notion.ts`

## 5. 영문 리포트 본문

### 5.1 같은 사고 페이지 본문 사용
- 체크 항목: 영문 리포트가 별도 DB가 아니라 같은 사고 페이지 본문에서 관리된다.
- 완료 기준: 사고 페이지 하나에서 고객 원문 속성과 영문 리포트 본문을 함께 확인할 수 있고, 별도 영문 리포트 DB 쓰기 경로가 없다.
- 검증 방법: 코드에서 영문 리포트 저장 경로를 확인하고, Notion에서 같은 페이지 본문을 직접 확인한다.
- 실패 시 먼저 볼 파일: `src/notion.ts`, `src/render.ts`

### 5.2 외부 제목과 기본 템플릿
- 체크 항목: 영문 리포트 제목과 템플릿 고정 항목이 문서와 일치한다.
- 완료 기준: 본문 또는 출력 경로에서 `Report a Save (Known or Suspected Finger Contact)` 제목과 `Phone`, `Email`, `Consent for Promotional Use` 항목이 확인된다.
- 검증 방법: 사고 페이지 본문 또는 출력 렌더 결과를 확인한다.
- 실패 시 먼저 볼 파일: `src/render.ts`, `src/notion.ts`

### 5.3 보완 업로드 재진입 가능
- 체크 항목: 영문 초안 작성 후에도 관리자 보완 업로드로 다시 들어갈 수 있다.
- 완료 기준: 사고 페이지에서 보완 업로드 진입 경로가 유지되고, 보완 후 다시 같은 사고 페이지 본문 수정 흐름으로 복귀할 수 있다.
- 검증 방법: 사고 페이지에서 보완 업로드 진입, 파일 업로드, 원래 페이지 복귀를 순서대로 확인한다.
- 실패 시 먼저 볼 파일: `src/render.ts`, `src/admin-upload.ts`, `src/index.ts`

## 6. 출력 / 발송 준비

### 6.1 같은 원본 본문 기준 출력
- 체크 항목: 웹뷰 또는 PDF 출력이 같은 영문 리포트 본문을 기준으로 생성된다.
- 완료 기준: 출력 내용이 사고 페이지 본문과 불일치하지 않고 별도 원문 저장소를 요구하지 않는다.
- 검증 방법: 사고 페이지 본문과 출력 결과를 같은 샘플 건으로 비교한다.
- 실패 시 먼저 볼 파일: `src/render.ts`, `src/notion.ts`, `src/index.ts`

### 6.2 발송 준비 완료 formula 판정
- 체크 항목: `발송 준비 완료(자동)`이 3개 체크값 조합으로만 결정된다.
- 완료 기준: `영문 검수 완료`, `첨부 최종 확인 완료`, `출력 확인 완료`가 모두 true일 때만 `발송 준비 완료(자동)`이 true다.
- 검증 방법: 3개 체크값 조합을 바꿔가며 formula 결과를 확인한다.
- 실패 시 먼저 볼 파일: `src/notion.ts`, `src/attachment-state.ts`

### 6.3 수동 발송 기본 유지
- 체크 항목: MVP에서 본사 전달 기본값이 수동 발송으로 유지된다.
- 완료 기준: 코드와 운영 흐름 어디에도 SMTP 완료 전 자동 발송 강제 경로가 없다.
- 검증 방법: 발송 관련 코드 경로와 설정값을 확인한다.
- 실패 시 먼저 볼 파일: `src/notion.ts`, `src/index.ts`, `src/mail.ts`

## 7. 운영 검증

### 7.1 손가락 사진 있음 write-back
- 체크 항목: 현재 첨부 중 손가락 사진 존재 여부가 체크박스에 반영된다.
- 완료 기준: 현재 첨부에 `첨부 유형=손가락 사진`이 1개 이상이면 `손가락 사진 있음=true`, 없으면 false다.
- 검증 방법: 손가락 사진 첨부 추가와 삭제를 각각 수행하고 사고 DB 체크박스를 확인한다.
- 실패 시 먼저 볼 파일: `src/attachment-state.ts`, `src/consumer.ts`, `src/admin-upload.ts`

### 7.2 첨부 최종 확인 완료 자동 해제
- 체크 항목: 첨부 변경 이벤트 발생 시 `첨부 최종 확인 완료`가 자동 false로 해제된다.
- 완료 기준: 새 첨부 추가, 유형 변경, 삭제, 휴지통 이동, 복구, FIFO 영구삭제, 마지막 손가락 사진 상실 중 하나가 발생하면 기존 true 값이 false가 된다.
- 검증 방법: 각 이벤트를 최소 1회씩 재현해 체크박스 변화를 확인한다.
- 실패 시 먼저 볼 파일: `src/attachment-state.ts`, `src/consumer.ts`, `src/admin-upload.ts`

### 7.3 미분류 고객 첨부 저장
- 체크 항목: 고객 웹폼 첨부는 분류 전 상태로 저장된다.
- 완료 기준: 고객 웹폼으로 생성된 첨부 DB 행은 `첨부 유형=null`이고, 관리자 업로드 생성 행은 null이 아니다.
- 검증 방법: 고객 제출 1건과 관리자 업로드 1건을 각각 만들어 첨부 DB를 비교한다.
- 실패 시 먼저 볼 파일: `src/consumer.ts`, `src/admin-upload.ts`, `src/notion.ts`

### 7.4 고객 화면 비노출 유지
- 체크 항목: 고객 화면에 내부 운영 상태가 노출되지 않는다.
- 완료 기준: 고객 웹폼과 완료 화면 어디에도 `첨부 업로드 상태`, `손가락 사진 있음`, `첨부 최종 확인 완료`, `발송 준비 완료(자동)`, Queue 상태가 없다.
- 검증 방법: 고객 화면 전체 텍스트와 DOM을 확인한다.
- 실패 시 먼저 볼 파일: `src/render.ts`, `src/index.ts`

## 8. 회귀 테스트 체크 항목

### 8.1 schema drift 회귀
- 체크 항목: live schema와 코드 상수 간 drift가 없다.
- 완료 기준: 속성명 누락, 타입 불일치, 허용값 불일치가 0건이다.
- 검증 방법: `node scripts/check-notion-schema.ts`와 `node scripts/check-allowed-values.ts`를 실행한다.
- 실패 시 먼저 볼 파일: `src/schema.ts`, `src/constants.ts`, `scripts/check-notion-schema.ts`, `scripts/check-allowed-values.ts`

### 8.2 고객 접수 회귀
- 체크 항목: 기본 접수 성공 흐름이 유지된다.
- 완료 기준: 정상 fixture 제출 시 고객은 성공 화면과 접수번호를 받고, 사고 DB 페이지와 기본 본문이 생성된다.
- 검증 방법: `node scripts/smoke-submit.ts` 실행
- 실패 시 먼저 볼 파일: `src/index.ts`, `src/normalize.ts`, `src/validate.ts`, `src/mapper.ts`, `src/notion.ts`

### 8.3 관리자 업로드 회귀
- 체크 항목: 관리자 보완 업로드 흐름이 유지된다.
- 완료 기준: 인증, 검색, 업로드, relation 연결, 유형 저장, 상태 반영이 한 번에 완료된다.
- 검증 방법: `node scripts/smoke-admin-upload.ts` 실행
- 실패 시 먼저 볼 파일: `src/admin-auth.ts`, `src/admin-search.ts`, `src/admin-upload.ts`, `src/notion.ts`

### 8.4 relation / R2 Key 회귀
- 체크 항목: 첨부 저장 후 relation과 `R2 Key`가 동시에 올바르다.
- 완료 기준: 첨부 DB 행마다 `사고건` relation이 존재하고 `R2 Key`는 `attachments/` 경로다.
- 검증 방법: 첨부가 있는 샘플 접수 1건과 관리자 업로드 1건을 조회해 DB 값을 확인한다.
- 실패 시 먼저 볼 파일: `src/consumer.ts`, `src/notion.ts`

### 8.5 write-back 회귀
- 체크 항목: 첨부 변경 후 `손가락 사진 있음`과 `첨부 최종 확인 완료`가 규칙대로 다시 계산된다.
- 완료 기준: 손가락 사진 추가 시 true, 마지막 손가락 사진 상실 또는 새 첨부 추가 시 `첨부 최종 확인 완료=false`가 재현된다.
- 검증 방법: 첨부 추가, 삭제, 유형 변경 시나리오를 순서대로 수행한다.
- 실패 시 먼저 볼 파일: `src/attachment-state.ts`, `src/consumer.ts`, `src/admin-upload.ts`
